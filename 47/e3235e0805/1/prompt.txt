## Ranked findings

### 1. `bd ready --claim` aborts (exit 2) when the ready pool exceeds the cap — default mode, common agent command — should-fix
`internal/storage/issueops/claim.go:206`

`ClaimReadyIssueInTx` copies the caller's filter, resets `claimFilter.Limit = 0`, but does **not** reset `MaxRows`/`MaxRowsSource`. It then calls `GetReadyWorkInTx`, which (post-PR) sizes `LIMIT` via `EffectiveSearchLimit(0, M) = M+1` and fires `EnforceMaxRowsCap` when it scans `M+1` rows.

Failure scenario: agent rig exports `BEADS_MAX_ROWS=100` globally (exactly the "CI/agent rig circuit breaker" use the design courts). In a repo with 150 unassigned-ready issues, `bd ready --claim` exits 2 with "too many rows" and cannot claim — even though claim's result is always ≤1 issue. Verified: `ready.go:210` wires `handleMaxRowsError` onto the claim path, and `embeddeddolt.ClaimReadyIssue` → `ClaimReadyIssueInTx` (the default local backend). The `ready_max_rows_test.go` suite only exercises `GetReadyWork`, never the claim path, so this is untested.

Suggested fix: mirror the `Limit = 0` reset — `claimFilter.MaxRows = 0; claimFilter.MaxRowsSource = ""` — since a single-issue claim can never legitimately be "too many rows." (Trade-off to weigh: that removes the bound on the internal ready-set scan; if that matters, set `claimFilter.Limit = filter.MaxRows` instead and still zero `MaxRows`, bounding the fetch without the erroring semantics.) This is a semantics call; flag for maintainer decision.

### 2. Cap is silently ignored in `--proxied-server` mode (flag never set *and* backend never enforces) — should-fix / document
`cmd/bd/list_proxied_server.go` (whole file — no `MaxRows`), `internal/storage/domain/db/issue_search.go:27` / `ready_work_union.go` (no `MaxRows`)

`runListCore` diverts to `runListProxiedServer` before `resolveMaxRows` is called (`list.go:487` returns early), so `filter.MaxRows` stays 0. Independently, the proxied backend (`issueSQLRepositoryImpl`, used by every `*_proxied_server.go` via `IssueUseCase()`) is a parallel search implementation that references only `filter.Limit`/`Offset`, never `MaxRows` — so even a threaded value would be ignored. Net: `bd list --max-rows 5 --proxied-server` parses the flag, ignores it, exits 0, prints all rows.

Mitigating: `--proxied-server` is marked `[EXPERIMENTAL]` (`init.go:2050`) and is opt-in per-workspace via config (`main.go:337 IsDoltProxiedServerMode`). Plain server mode and embedded mode both route through capped `issueops`, confirmed by the passing `doctor_max_rows_test.go` server-mode gate. Silently no-op'ing a *safety* flag is still the worst option — either reject `--max-rows` under proxied-server, or track the gap alongside the already-documented `GetReadyWorkWithCountsInTx` (ready --json) hole in the same commit message. Same pattern, same treatment.

### 3. `--limit N --max-rows N` (equal) errors on the fetch-one-extra probe row — nit
`cmd/bd/list.go:42` (`withFetchOneExtra`) interacting with `EffectiveSearchLimit`

`withFetchOneExtra` bumps `Limit` to `N+1`; `EffectiveSearchLimit(N+1, N)` returns `N+1` (limit &gt; maxRows), the SQL returns `N+1` rows when more than N match, and `EnforceMaxRowsCap(N+1, N)` fires. But the row set actually *delivered* after Limit-trim would be exactly `N` — which does not exceed the cap. So `bd list --limit 5 --max-rows 5` errors where plain truncation was expected. Only reachable at `N == M` with `&gt;M` matches; `N&lt;M` and `N&gt;M` behave correctly. Consider comparing the cap against the delivered count, or documenting that a fetch-probe row counts toward the cap.

### 4. `os.Exit` in the cap error path skips `metrics.CloseAndFlush()` — nit (telemetry only)
`cmd/bd/max_rows.go:75` (`fatalMaxRowsUsage`), `cmd/bd/max_rows.go:130` (`handleMaxRowsError`)

Both call `os.Exit()` directly from inside a RunE body. `main.go:1685` flushes queued metrics only after `ExecuteC()` returns, and the established pattern for direct-exit paths (`CheckReadonly`, `errors.go:136`) calls `metrics.CloseAndFlush()` first. These two don't, so queued events for the invocation are stranded. `fatalMaxRowsUsage` faithfully mirrors the *old* `FatalError` (verified: it also didn't flush), so this isn't a new regression — but `handleMaxRowsError` runs deep in the command after events have queued, and main's direction was to stop `os.Exit`-in-RunE precisely to preserve cleanup. `handleMaxRowsError` already sits on a path where the caller does `return HandleError(...)`, so it could `return &amp;exitError{Code: 2}` instead of `os.Exit(2)` — simpler, flushes metrics, and aligns with the RunE philosophy the file comment claims to honor.

### 5. `bd graph --all` cap is per-status, not total — nit (acknowledged)
`cmd/bd/graph.go:436` — the cap is threaded into each of three per-status `SearchIssues` calls independently, so a total of up to `3*(cap)` rows can pass without firing. The comment says so explicitly; acceptable for a circuit breaker but inconsistent with the "rows returned" mental model. Same for `bd dep tree` (`dep.go:1096`), which enforces post-hoc on `len(tree)` after building the full tree — the walk isn't actually bounded, only the reporting.
