The PR backlog is not one problem. It is three queues sharing one GitHub view: release-safety work around Dolt/JSONL/hooks/storage, architecture work around storage/server/plugin boundaries, and community throughput work containing old PRs, dependency bumps, docs, integrations, and features.

Treating all 138 open PRs as one review pile is what makes the situation feel impossible. Split the pile into lanes, land narrow release-critical fixes, convert obsolete or architecture-blocked PRs into tracking issues, and preserve contributor credit when absorbing useful tests or repros.

3340 / 3626 / 3598 / 3599

#3340 reports a real class of deadlock: an internal git commit runs while a parent bd process still owns the embedded Dolt lock; the pre-commit hook runs; the hook shells out to bd export; the child process tries to open the same embedded store.

#3626 fixes this with --no-verify in commitBeadsConfig and adds a cgo embedded regression test. #3599 is an earlier duplicate for #3598 with useful hostile-hook unit tests, commit-msg coverage, and idempotence coverage.

Recommended action: land one combined fix, not two competing PRs. Use #3626 as the fresh mergeable vehicle or a maintainer branch, add useful #3599 unit-test coverage, credit both contributors, then close duplicates with an explanation.

Local repro note: a pure-Go/Linux build did not reproduce the hard hang on current main or #3626. That is not decisive because Linux normally has GNU timeout, and the hard report is stock macOS without timeout on the embedded/cgo path. Current upstream/main still has git commit -m without --no-verify in cmd/bd/sync_remote.go.

3354 / 3356 / 3937 / 3940

#3354 reported massive tmp_pack_* growth. #3356 found that bd init registered plain Git origins as Dolt remotes even when no refs/dolt/data existed. #3357 fixed that and should be treated as correct safety work.

#3937 restates the policy: do not treat a plain Git origin as existing Dolt history during bootstrap/pull; do let explicit bd dolt push adopt Git origin as the target when no Dolt remote exists. #3940 implements this narrower push-time adoption and preserves the #3356/#3354 protection.

Recommended action: keep #3940 as a release candidate only after a scenario matrix covers no-origin init, plain-origin init, push-time adoption, bootstrap/pull gating on refs/dolt/data, server/shared-server behavior, and JSONL not being treated as canonical sync.

Plugin Architecture

The plugin proposal is strong as a design spine: providers over MCP, automations as sandboxed WASM, explicit grants, content-addressed lockfiles, storage data plane kept in-tree, and a frozen Cobra tree.

It should not become a reason to stall narrow safety fixes. Use it to defer or reroute integration and extension PRs, not to block current core bug fixes.

Charter Assessment

docs/PROJECT_CHARTER.md is directionally right. It states core scope, orchestration boundary, storage boundary, schema boundary, integration boundary, and review posture.

The missing piece is an operational PR intake rubric. The charter says what belongs in Beads; it does not yet tell an agent how to triage 130 open PRs quickly and safely.

Add a short decision tree near PR_MAINTAINER_GUIDELINES.md: release blocker, narrow landable, fix-merge, absorb as issue, plugin/integration candidate, retire. Then point AGENTS.md at both the charter and the rubric.

Backlog Reduction Path

Lane 1: Release Stabilization

Only accept release-critical regressions, data safety fixes, deadlocks, install/build failures, severe performance fixes, or Julian's explicitly requested orchestration support.

Lane 2: Architecture Hold

Label and pause broad storage/server/daemon/plugin-shaping PRs. Ask for tracking issues and repro evidence until storage/server work and the plugin architecture have enough shape to review against.

Lane 3: Backlog Harvest

For older PRs, preserve the useful part: repro, regression test, use case, small patch, or architecture note. Then close or convert. The goal is to reduce count without losing signal.
