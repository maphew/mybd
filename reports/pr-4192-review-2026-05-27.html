Verdict

merge-fix (lean toward merge with a follow-up beads issue).

The change is a correct, narrow P0 fix that eliminates the corruption mode reported in #3722. It is safe to land for v1.0.4 triage. However, it implements option 1 (detect + refuse) from the upstream issue, while the codebase had recently shipped option 3 (detect + follow symlink) via c7615e407 fix(setup): preserve symlinks in atomicWriteFile. That difference is a behavior regression for the most common deployed configuration (CLAUDE.md -> AGENTS.md), where bd setup claude will now silently no-op rather than write integration text into the real AGENTS.md. The corruption is fixed, but users who relied on the previous follow-through behavior must now invoke setup explicitly against the link target.

Recommendation: merge as-is to unblock the P0, then file a beads follow-up to either restore option 3 safely or document the new contract clearly in bd setup --help and the integration READMEs.

Why it exists

Upstream issue #3722 (yakshaver / Corey Klaasmeyer) reports that bd init against a symlinked CLAUDE.md -> AGENTS.md produces a git index entry with mode 120000 whose blob is the multi-line string "AGENTS.md\n\n<!-- BEGIN BEADS INTEGRATION -->...\n". macOS APFS caps symlink targets at ~1023 bytes, so the resulting repo is unrecoverable on macOS: git rebase, git merge, fresh clones, and branch-switch all fail with unable to create symlink CLAUDE.md: File name too long. This affects v1.0.3 (Homebrew) and earlier; the linked beads mybd-l4r flagged it P0 for v1.0.4 triage.

The reporter offered three fix strategies (refuse / flip-to-regular-file / follow-target). This PR implements strategy 1 (refuse), which is the most conservative.

What it changes

cmd/bd/setup/agents.go (lines 91-114 of new content):

  • Drops the utils import (no longer needed after removing ResolveForWrite).
  • Replaces the utils.ResolveForWrite(env.agentsPath) + read-resolved-path block with an os.Lstat check that:
    • Returns nil (silent success after a warning) when env.agentsPath is a symlink.
    • Returns the lstat error (and logs it) for non-IsNotExist errors.
    • Falls through to the existing flow when the path is a regular file or does not exist.
  • Reads os.ReadFile(env.agentsPath) directly (no longer the resolved path) for the existing-content read. The atomicWriteFile call further down still goes through utils.ResolveForWrite for its own write, but since the early symlink check returns before that, that resolve never fires.

cmd/bd/setup/agents_marker_test.go::TestInstallAgentsSymlinkSafety (lines 316-370):

  • Test scenario inverted. Previously it seeded a full-profile target and asserted that a minimal install through the symlink preserved full-profile content in the target. Now it seeds arbitrary non-managed content ("# Existing target content\n") and asserts that:
    • The link remains a symlink (Lstat).
    • The target's content is byte-equal to the seed (no managed-section injection).
    • The "is a symlink; skipping managed section injection" warning appears in stdout.

Strengths

  • Directly closes the corruption path described in #3722. Refusal-with-warning is the safest possible behavior; it cannot break a working repo.
  • Minimal surface area. One file plus one test; no public API changes.
  • Test was rewritten in lockstep with the behavior change. The new assertions match the new contract and are explicit (link mode preserved, target untouched, warning emitted).
  • The PR description honors the project convention ("Why" + "What changed" + drift check + validation + agent signing).
  • installAgents callers (claude.go, mux.go, gemini.go, factory.go, opencode.go) already treat its error as non-fatal, so even an unexpected lstat failure no longer aborts the higher-level integration.
  • No storage-engine boundary concerns: the change is purely in cmd/bd/setup/ and touches no storage drivers, locks, or engine introspection.

Concerns

  1. Behavior regression vs. recently-shipped option 3. cmd/bd/setup/utils.go:23 and cmd/bd/setup/utils_test.go:91 TestAtomicWriteFile_PreservesSymlink document that the codebase was deliberately set up to write through symlinks via ResolveForWrite (commit c7615e407 fix(setup): preserve symlinks in atomicWriteFile). For users with the canonical CLAUDE.md -> AGENTS.md pattern, running bd setup claude now leaves AGENTS.md without the integration block, where previously it would have written it. The PR description and warning message do not mention this regression or what the user should do next. Suggest: extend the warning to name the resolved target and recommend the corrective command, for example: 
    Warning: CLAUDE.md is a symlink to AGENTS.md; skipping.
Re-run with the target file (e.g. 'bd setup claude --agents-file AGENTS.md')
to install integration there.
    This costs nothing and turns a silent regression into a directive.
  2. Symlink-safe atomicWriteFile test is now stranded. cmd/bd/setup/utils_test.go:91-124 TestAtomicWriteFile_PreservesSymlink still asserts symlink-preserving write semantics for atomicWriteFile. That utility is still used by callers other than installAgents (claude.go:55, codex.go:65, gemini.go:55, aider.go:165-175, junie.go:128/139, mux.go:109, cursor.go:65, agent_skill.go:40). If any of those caller paths can be reached with a symlink-as-target (e.g. a user creates .claude/settings.json as a symlink), they will still follow the symlink and could in principle reproduce a similar corruption mode for those artifacts. The PR only locks down installAgents. Suggest filing a follow-up to audit other atomicWriteFile call sites for symlink-as-target risk, or to apply the same Lstat guard in atomicWriteFile itself (with an explicit opt-in for callers that genuinely want pass-through).
  3. Lost code-coverage on the lstat error branch. Codecov flags cmd/bd/setup/agents.go patch coverage at 75% with 1 missing and 1 partial line. The uncovered branch is almost certainly the else if err != nil && !os.IsNotExist(err) arm. Worth a one-line test (e.g. set env.agentsPath to a path whose parent directory is mode 0000, or a path that traverses a non-directory) to lock down behavior. Not a blocker.
  4. Warning is on stdout, not stderr. cmd/bd/setup/agents.go new line ~98 emits the symlink warning via fmt.Fprintf(env.stdout, "Warning: ..."). Other diagnostic messages in the same file use env.stderr (e.g. line 103-104 in the new code: fmt.Fprintf(env.stderr, "Error: failed to inspect ...")). This is a small inconsistency; the test asserts against stdout, so it was a deliberate choice — but a Warning: line on stdout will mix with success messages in piped/captured output. Worth aligning to stderr (and updating the test) for cleanliness. Not blocking.
  5. The early return path skips the "integration installed" tail. Lines 154-163 print final summary including File:, readHint, and docsURL. When the symlink branch returns early on line 96, the user gets only "Installing X integration..." plus the warning, no follow-up. That is correct for the no-op case, but claude.go:212-214 then unconditionally prints "✓ Claude Code integration installed" and a restart hint, which is misleading when the integration text never landed. Suggest the symlink branch return a sentinel (e.g. errAgentsSymlinkSkipped) that claude.go/mux.go can detect and adjust their final messaging. Cosmetic; can be a follow-up.
  6. No coverage for the read-but-not-symlink path being unchanged. The diff removed the ResolveForWrite indirection in the read step (now reads env.agentsPath directly). For non-symlink paths, behavior is identical, but no test exercises this regression boundary explicitly. Existing tests (TestInstallAgentsPreservesFullProfile, TestLegacyToNewMigrationViaInstall) will catch a complete break, but a targeted assertion that read-resolved-path is no longer used (e.g. that a symlink in the containing directory does not affect the read) would be ideal. Not blocking.

Storage-driver-boundary check

No violations. The PR touches only cmd/bd/setup/. There are no storage engine calls, no flock-style locking, no Dolt or SQLite introspection. Per the beads-storage-driver-roadmap memory, this PR is in the clear.

Backwards compatibility & migration

  • Forward: users who had no symlinks see no behavior change. Users who had symlinks but no integration installed lose the ability to install integration by pointing setup at the symlink — they must point at the target file. Worst case is a confused user who has to re-run.
  • Backward: existing valid managed sections written through this code path are not modified. Existing corrupt repos (mode 120000 + multi-line blob) are not auto-repaired; the manual git update-index --cacheinfo 100644,... recipe in #3722 still applies. Worth mentioning in release notes.

Recommendations

  1. Land this PR for v1.0.4. It correctly stops the corruption. The behavior trade-off is acceptable for a P0 fix.
  2. Before merge (cheap polish; do as in-place maintainer edits on the contributor branch per PR_MAINTAINER_GUIDELINES.md since the contributor is the same user):
    • Expand the warning to name the resolved target via os.Readlink(env.agentsPath) and suggest the corrective command. (Concern 1.)
    • Optionally move the warning to env.stderr for consistency, updating the test. (Concern 4.)
  3. File follow-up beads issues (do not block):
    • Audit atomicWriteFile callers for symlink-as-target reachability and decide whether to push the Lstat guard down into the utility. (Concern 2.)
    • Add a test for the lstat error branch in installAgents. (Concern 3.)
    • Decide whether claude.go/mux.go final messages should detect the symlink-skip case and adjust their summary. (Concern 5.)
    • Add a "repair recipe" snippet to release notes / docs for users already stuck with mode-120000 multi-line CLAUDE.md blobs (the exact git update-index --cacheinfo 100644,<sha>,CLAUDE.md recipe from the issue).
  4. On merge: close gastownhall/beads#3722 with a note linking this PR, and close beads mybd-l4r per the local triage entry's exit criterion.
