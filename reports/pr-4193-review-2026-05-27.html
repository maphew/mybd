Verdict

merge.

Tiny, surgical, test-only diff that fixes a real parallel-run flake in TestResolveWhereBeadsDir_UsesInitializedDBPath by adopting the same isolation pattern already used by its two sibling tests in the same file. The fix is correct against the actual fallback chain in cmd/bd/main.go:282-311 (selectedNoDBBeadsDir), CI is fully green, Codecov reports complete patch coverage, and there is no production code change. No storage-driver-boundary concerns (test-only). I would land this as-is.

The only nit worth mentioning is that this test still does not reset the --db Cobra flag the way the sibling tests do (where_test.go:53-63 and where_test.go:98-108), but it is safe here because the test calls resolveWhereBeadsDir(nil) directly and both flag-checking branches in selectedNoDBBeadsDir are guarded by cmd != nil (main.go:283, 287). Worth a one-line follow-up note for consistency, not a blocker.

Why it exists

Linked beads issue mybd-b8i (closed; external gh-pr-4193) describes the flake:

When running CGO_ENABLED=1 go test -tags gms_pure_go -count=3 -p 4 ./cmd/bd/, TestResolveWhereBeadsDir_UsesInitializedDBPath fails with: resolveWhereBeadsDir(nil) = '/home/<user>/dev/mybd/.beads', want '/tmp/TestResolveWhereBeadsDir_UsesInitializedDBPath.../.beads'. The test sets dbPath=dbDir and chdirs into a t.TempDir, but resolveWhereBeadsDir falls through to beads.FindBeadsDir() which walks up from cwd to the project root.

The triage entry explicitly distinguishes this from the drift-test chdir race (a separate failure mode): the root cause here is global env vars (BEADS_DIR / BEADS_DB / BD_DB) and/or other globals leaking from sibling tests, hitting the where.go fallback chain at cmd/bd/main.go:270-298. Under -p 4, parallel cmd/bd test binaries share that process state; whichever test ran most recently could leave BEADS_DB set, which then short-circuits selectedNoDBBeadsDir (main.go:291) before the dbPath branch at line 300 is reached, returning the surrounding repo's .beads/ instead of the test's tempdir.

Discovered while fixing mybd-j2s (PR #3747).

What it changes

cmd/bd/where_test.go::TestResolveWhereBeadsDir_UsesInitializedDBPath (lines 115-157 of current file; net +7 / -3 lines):

  • Drops the manual save/restore of dbPath (old lines 116, 119) in favour of saveAndRestoreGlobals(t) (defined in cmd/bd/test_helpers_pure_test.go:141), which snapshots dbPath, store, storeActive, and the export* flags atomically and registers a t.Cleanup for restoration on panic or normal exit.
  • Adds ensureCleanGlobalState(t) (test_helpers_pure_test.go:106), which calls resetCommandContext() so accessor functions fall back to legacy globals rather than carrying stale cmdCtx state from another test.
  • Keeps the existing originalCmdCtx := cmdCtx / deferred restore (lines 117, 120) for explicit cmdCtx rollback — this is redundant with ensureCleanGlobalState's resetCommandContext(), but it is defensive and matches what the test was already doing for cmdCtx.
  • Adds three t.Setenv calls clearing BEADS_DIR, BEADS_DB, and BD_DB (new lines 153-155 in the diff). These cover exactly the env vars that selectedNoDBBeadsDir reads at cmd/bd/main.go:291, 295, and 305 and which findOriginalBeadsDir / beads.FindBeadsDir() consult further down.
  • Replaces the bare assignment dbPath = dbDir with the canonical setter setDBPath(dbDir) (cmd/bd/context.go:153). setDBPath updates both the legacy global and cmdCtx.DBPath when cmdCtx != nil, so the test is consistent regardless of whether the resetCommandContext landed first.

Nothing else moves: the workspace setup, the os.Chdir(cwd), the metadata save, and the final assertion all stay identical.

Strengths

  • The fix reuses existing, well-documented helpers (saveAndRestoreGlobals, ensureCleanGlobalState, setDBPath) rather than inventing new abstractions. The helpers' docstrings in test_helpers_pure_test.go:126-140 and context.go:71-76 explicitly describe this use case.
  • It matches the isolation pattern already adopted in the two sibling tests at the top of the same file (TestResolveWhereBeadsDir_FallsBackToFindBeadsDir at lines 17-68, TestResolveWhereBeadsDir_ReturnsEmptyWithoutWorkspace at lines 70-113). After this PR, all three TestResolveWhereBeadsDir_* tests use the same skeleton — a clear consistency win.
  • The env vars cleared (BEADS_DIR, BEADS_DB, BD_DB) exactly correspond to the three env reads in the selectedNoDBBeadsDir fallback chain (main.go:291, 295, 305); the fix targets the root cause rather than masking symptoms.
  • Using t.Setenv instead of os.Setenv ensures Go's test framework restores prior values on exit, even if the test panics.
  • Validated by the PR description: -count=10 against the target test plus -count=3 -p 4 ./cmd/bd/ against the whole package, the same invocation that reproduced the flake.
  • CI is fully green across 41 checks including macOS, Windows-smoke, Embedded Dolt (20 shards), Differential Regression, and all 5 cross-version upgrade smokes. Codecov: complete patch coverage.
  • Net code reduction in the local save/restore boilerplate (one fewer local variable; the helper does it).

Concerns

  1. --db flag is not reset. The two sibling tests (where_test.go:53-63, where_test.go:98-108) explicitly capture and clear rootCmd.PersistentFlags().Lookup("db") to defend against another test having called something like cmd.PersistentFlags().Set("db", ...). This test does not. It happens to be safe today because the call site is resolveWhereBeadsDir(nil) and the two --db branches in selectedNoDBBeadsDir (main.go:283, 287) both require cmd != nil. But if a future refactor changes the test to pass a real *cobra.Command (for example to assert behavior under an explicit --db flag), the omission becomes a latent flake source. Suggest a one-line follow-up to add the same flag-reset block for consistency. Nit, not a blocker.
  2. originalCmdCtx save/restore is now redundant with ensureCleanGlobalState. ensureCleanGlobalState(t) (lines 71-72, 116 in current file) calls resetCommandContext() which sets cmdCtx = nil. The remaining originalCmdCtx := cmdCtx / defer { cmdCtx = originalCmdCtx } pair (kept in the diff) was meaningful before this PR. After the PR, ensureCleanGlobalState already nukes cmdCtx; restoring whatever was there at function entry is harmless but cosmetic. The two sibling tests carry the same redundancy (where_test.go:21-24, 74-77), so the diff is consistent with house style — but a future cleanup could drop the manual cmdCtx dance across all three tests. Cosmetic, no action needed.
  3. ensureCleanGlobalState calls resetCommandContext after saveAndRestoreGlobals has already snapshotted globals. Order matters subtly: saveAndRestoreGlobals(t) snapshots dbPath, store, etc. first; then ensureCleanGlobalState(t) resets cmdCtx. The cleanup order in t.Cleanup is LIFO, so the snapshotted globals will be restored after any later setDBPath calls — correct. No actual bug, but worth flagging that the two helpers' interaction depends on call order. The two sibling tests use exactly the same order, so the convention is already established.
  4. Validation strings in the PR body are escaped (\n not real newlines). The PR description renders with literal \n sequences in the "Validation" section. Cosmetic only; does not affect the code change or CI. The scripts/gh-body-lint referenced in AGENTS.md would have flagged this. No action required from a review standpoint.

No violations. This is a pure test-only change in cmd/bd/where_test.go. It touches no storage drivers, no flock-style locking, no Dolt/SQLite engine introspection, and no storage package internals. The helpers it pulls in (saveAndRestoreGlobals, ensureCleanGlobalState, setDBPath) all already live in cmd/bd/ and operate on CLI-layer globals, not on storage internals.
