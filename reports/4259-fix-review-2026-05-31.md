Date: 2026-05-31

## Executive summary

**The core fix is correct, and the work is not confused.** The parts that carry the weight — the deterministic id derivation, the idempotent convergence migration, the upgrade backfill, the prevention gate, and the skew detector — all do what their descriptions claim, and several genuinely subtle hazards are handled correctly (server-mode remote-sync ordering, `dolt_ignore`'d wisp tables, source-rename cascade, the git-protocol pull path).

Why it *reads* as a confused sprawl: one bug is split across three PRs, each wrapped in a long, jargon-dense write-up, and two of the write-ups oversell what they deliver. The size is mostly real — but **~29 of the 57 changed files are tests**, which is a quality signal, not noise.

## The incident, in one paragraph

Beads syncs its issue DB across machines like git syncs code. v1.0.5's migration `0043` gave the `dependencies` table a surrogate primary key `id CHAR(36) NOT NULL DEFAULT (UUID())`. `UUID()` is **per-clone random**, so when two independently-synced clones each upgraded in place, the same logical edge got two different primary keys (and a runtime-conditional migration could even leave the two clones on structurally different schemas at the same recorded version). Dolt then refuses the merge —

```
Error 1105: cannot merge because table dependencies has different primary keys in its common ancestor
```

— with no `bd`-level recovery. It triggers silently on first launch after `brew upgrade`, in the "independent peers around a central remote" topology that `bd dolt push/pull` is designed for. That was the v1.0.5 release-blocker.

## PR #4266 — deterministic dependency primary keys (the cure)

**What it does.** Replaces the random surrogate key with one derived from the edge's natural identity, so the same edge gets the same id on every clone and merges cleanly.

**Verified correct:**

1. **Deterministic derivation** (`internal/storage/depid/depid.go`):
   ```go
   func New(issueID, target string) string {
       return uuid.NewSHA1(Namespace, []byte(issueID+sep+target)).String()
   }
   ```
   A standard RFC-4122 v5 (SHA-1) UUID over a fixed namespace (`bdd74eb9-…`, hardcoded forever) and the key `issueID + 0x1f + target`. The separator is ASCII Unit Separator, which cannot appear in an id or target, so the encoding is unambiguous. This is the textbook-correct way to make a surrogate key clone-stable. ✓

2. **Migration `0050` is unconditional and idempotent** (`…/migrations/0050_…up.sql`): it drops the random default (guarded on `COLUMN_DEFAULT IS NOT NULL`) and re-asserts the three `uk_dep_*` unique keys with `CREATE UNIQUE INDEX IF NOT EXISTS`. It does **not** probe whether `0043` ran a particular branch — it just asserts the canonical end state, which is exactly what defuses the conditional-DDL divergence hazard. `0043` is also hardened to drop the default at its end (belt-and-suspenders; `0050` is load-bearing). ✓

3. **The critical cross-path consistency check.** The one place a silent re-fork could still hide is if an *insert* path derived the id over a differently-encoded target than the *backfill* does. I checked both:
   - Insert (`internal/storage/issueops/dependencies.go:218`): `depid.New(dep.IssueID, dep.DependsOnID)`, and the same `dep.DependsOnID` is written into the typed target column.
   - Insert (`internal/storage/domain/db/dependency.go:114`): identical — `depid.New(dep.IssueID, dep.DependsOnID)`.
   - Backfill (`internal/storage/schema/dep_id_backfill.go`): keys over `COALESCE(depends_on_issue_id, depends_on_wisp_id, depends_on_external)` — i.e. the value that was *stored*, which is the value the insert keyed on.

   So a freshly-inserted edge and a migrated-then-backfilled edge for the same logical dependency compute the **same** id. The insert/backfill paths cannot silently disagree. This is the property the whole fix rests on, and it holds. ✓

   External targets carry the `external:` prefix inside `dep.DependsOnID` and that full prefixed string is both stored and keyed, so external vs issue targets don't collide. Rename paths re-derive too (target rename = delete+reinsert; source rename via `rekeyDependencySourceInTx` after the FK `ON UPDATE CASCADE`, since the cascade moves `issue_id` but does not recompute the surrogate).

4. **Backfill is idempotent and safe** (`rekeyDependencyIDs`): rewrites only rows whose id differs from `depid.New(...)`, skips a missing table/column, skips a malformed no-target row (leaves it for `bd doctor`), runs only on a migration pass. ✓

5. **Round-2 review fixes are real.** The git-protocol pull path previously returned before the auto-resolver ran (so the `git+https` topology #4259 actually uses never got resolution) — now routed through `finishCLIPull → autoResolveConflictsAfterCLIPull` on the pull's own branch connection. And `dependencyConflictsAreAuditOnly` now verifies `issue_id` + resolved target from `dolt_conflicts_dependencies` instead of trusting the primary key, so a stale id can't collapse two different edges. Both are correct and were genuine bugs the first round missed.

**Residual concerns (minor):**

- **The key omits `type`.** `depid` keys on `(issue_id, target)`, but the three `uk_dep_*` unique keys are each on a *different* target column, so the deterministic PK is fractionally narrower than their union. A collision needs the same string to appear as both `depends_on_issue_id` and `depends_on_wisp_id` for one source issue (external is prefix-namespaced, so it's safe). That requires issue-id/wisp-id namespace overlap — very unlikely — and it would surface as a **loud** PK-collision insert failure, not a silent fork. Negligible, but worth a reviewer's nod and one sentence in the `depid` doc.
- **Suggested test (belt-and-suspenders):** an explicit assertion that an edge created via the insert path and the *same* edge arrived-at via `0043`-default-then-`rekeyDependencyIDs` produce byte-identical ids. The code is consistent (verified above), but this is the exact invariant a future refactor could break silently, so pinning it is cheap insurance. The existing `TestDependencyDeterministicIDMergesAcrossBranches` covers insert-vs-insert, not insert-vs-backfill.

## PR #4268 — prevention gate (the stop sign)

**What it does** (`internal/storage/schema/remote_migrate_gate.go`). Before auto-applying pending migrations on a read/write open, `CheckRemoteMigrateGate` refuses when the DB **has a recorded version AND has pending migrations AND has a remote configured**, returning a `RemoteMigrateGateError` that tells the operator to either migrate as the designated machine (`BD_ALLOW_REMOTE_MIGRATE=1`) or adopt the remote's copy (`bd bootstrap`). No-op for fresh DBs (`current == 0`), already-current DBs (no pending), no-remote DBs, and the escape hatch.

**Verified correct:**

- Gate condition is exactly as described; wired into **both** `DoltStore.initSchema` (server) and `EmbeddedDoltStore.initSchema` (embedded). Embedded coverage matters because the original report was embedded mode — an earlier draft only gated server mode and review caught it.
- The server-mode subtlety is handled well: a freshly auto-started `dolt sql-server` has an empty `dolt_remotes` table until the post-open sync (GH#2315), so a SQL-only check would miss the remote on the first write open after upgrade. `CheckRemoteMigrateGateWithRemoteCheck` adds an on-disk `.dolt/config` probe, invoked **only** when pending>0 and the SQL table shows no remote, so the subprocess probe stays off the common path. This is careful, not confused.
- `anyDoltRemoteConfigured` treats a missing `dolt_remotes` table as "no remote" so a missing system table can never wedge every open. ✓

**The real concern — it's a blunt instrument.** The gate fires for **any** remote-backed DB with a pending migration, including a single-machine user who merely pushes beads to GitHub (the documented mybd sync path). That user can't fork anything — there's no second clone — yet every migration-bearing upgrade halts them with an "unrecoverable fork" message and tells them to set an env var. Two follow-on risks:

1. **Cry-wolf / habituation:** training solo users to reflexively run `BD_ALLOW_REMOTE_MIGRATE=1` means the bypass becomes muscle memory exactly when a real second machine later appears.
2. The PR itself names the better design — "optionally read the remote's `schema_migrations` ledger to auto-detect you'd be the second migrator and refuse outright" — and defers it. That smarter gate fires only when migrating is *actually* dangerous (remote already migrated).

Crucially, **#4270 builds the exact primitive that smart gate needs** (reading the remote's recorded migrations from the cached ref, no network). As shipped, the two PRs stop one step short of combining into the gate that would fire only when it should. See recommendations.

## PR #4270 — per-migration content hash + `bd doctor` skew (the smoke detector)

**What it does.** Adds `content_hash CHAR(64)` (SHA-256 of each migration's file content) to `schema_migrations`, and a read-only `bd doctor` check that warns when this DB and its remote applied different content for the same migration version.

**Verified correct:**

- `ContentHashSkew(local, remote map[int]string)` (`internal/storage/schema/content_skew.go`) is a pure function: returns versions present on both sides with non-empty, differing hashes, sorted; ignores `NULL`/missing. Clean and reusable. ✓
- The doctor check (`cmd/bd/doctor/migration_skew.go`) compares against the **cached** remote-tracking ref via `… AS OF CONCAT('remotes/', ?, '/', ?)` — **no network fetch** — and degrades cleanly when there's no remote, no cached ref, or no hashes. Read-only; does not gate push/pull. ✓
- Column added to fresh DBs via `bootstrapSQL` and idempotently to existing DBs via `ensureContentHashColumn` (INFORMATION_SCHEMA-guarded `ALTER`). Deterministic column + hashes, unchanged PK ⇒ two clones upgrading independently still reach an identical `schema_migrations` and merge cleanly (the opposite of the `DEFAULT (UUID())` problem). Uses **no numbered migration**, deliberately, to avoid colliding with #4266's `0050`. ✓

**The honesty gap.** Already-applied rows keep a `NULL` hash ("their content is not re-read"). That means on the databases that *already* hit #4259, migration `0043`'s row has no hash — so this detector **cannot see the very fork that prompted it**. It only catches *future* divergent migrations.

The `NULL`-for-existing choice is actually defensible: you can't know what content was *historically* applied, only what the current binary thinks migration N is, so backfilling would write matching hashes on both clones and give false confidence. Leaving `NULL` is the honest behavior. But it makes the PR title — "reporter fix No.2 for #4259" — oversell: this is forward-looking insurance against the *class*, not a fix or even a detector for the incident. Worth restating plainly in the PR and not banking on it for the current event.

## Cross-PR concerns

**These are not independently landable in arbitrary order.** They overlap on shared files and will conflict:

- `internal/storage/dolt/schema_skew_test.go` — edited by all three (+26/−3 each), almost certainly the same region.
- `internal/storage/schema/lock_test.go` — #4266 (+14) and #4270 (+11/−2).
- `internal/storage/dolt/store.go` — #4266 (+214/−32) and #4268 (+10).
- Migration numbering: #4266 owns `0050`; #4270 deliberately took no number to avoid it. So #4266 must be the reference point.

**Order:** land #4266 → rebase #4268 → rebase #4270. Re-run the suite after each rebase (the shared test files are where a bad merge would hide).

## Recommendations

1. **Land #4266 first.** It's the real fix, it's correct, and it owns the migration number. Add the insert-vs-backfill equality test before or just after merge; add one sentence to `depid`'s doc noting `type` is intentionally excluded and the (negligible, loud-failing) cross-typed-column collision.
2. **Reconsider shipping #4268 and #4270 separately.** Combine them: use #4270's cached-remote-ledger read to make #4268's gate fire **only** when the remote is already at/ahead of the binary (i.e. you'd be the second migrator). That converts a blunt, cry-wolf gate into one that's silent for solo users and loud exactly when a fork is imminent. If a blunt gate must ship now for the release, at minimum soften the solo-user path (detect "no divergence possible" / single-clone) so the scary message is reserved for genuine risk.
3. **Re-title / re-scope #4270** away from "fix No.2" toward "forward-looking divergence detection," and state explicitly that it does not detect `0043` on already-migrated DBs.
4. **CI gap (from the #4259 plan, not yet in these PRs):** the two-clone-migrate-then-merge and two-clone-same-edge-then-merge integration recipes. #4266 has branch-level merge tests, but the end-to-end "two independent clones around a remote" recipe — the actual incident topology — is the regression test that would prove the whole class closed.
