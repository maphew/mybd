Verdict

ready-to-merge (with one optional follow-up). This is an easy win.

The PR is small, focused, and addresses a real reproducible P0 bug in v1.0.3 / current main. The implementation reuses existing storage-driver primitives (doltAddAndCommit, Commit, isDoltNothingToCommit) and mirrors a well-established pattern already used in PullFrom (GH#2474). It lives entirely inside internal/storage/dolt/, which is the correct side of the storage-driver boundary — no beads-side code is being asked to know anything new about Dolt. A regression test is included. CI is green (codecov is only flagging that the new lines aren't exercised by non-integration runs, which is expected for tests gated by //go:build integration).

Why it exists

bd federation add-peer writes a row into federation_peers but does not DOLT_COMMIT it. Any subsequent bd federation sync triggers DOLT_FETCH -> DOLT_MERGE, and the merge refuses to proceed with the working set dirty:

merge failed: local changes would be stomped by merge: federation_peers

The reporter (dlwiii) provided a clean reproducer plus a worrying side-effect note: workarounds involving killing the bg dolt sql-server and running dolt gc apparently caused silent loss of two locally-created issues. So the fix is on the P0 path because it removes the user's incentive to hand-roll dangerous workarounds.

The current Sync also did not perform the pre-merge auto-commit that PullFrom (internal/storage/dolt/federation.go:57-66) already does. So even if add-peer had committed cleanly, any other source of working-set churn between add-peer and sync (e.g. concurrent issue writes) would have hit the same wall.

What it changes

  1. internal/storage/dolt/credentials.go:308-311 — after AddFederationPeer upserts the row and registers the Dolt remote, it now calls s.doltAddAndCommit(ctx, []string{"federation_peers"}, "federation: add peer "+peer.Name). The commit is scoped to the single table, matching the GH#2455 pattern used by dependencies.go:55,88 and issues.go callsites.
  2. internal/storage/dolt/federation.go:294-303 — Sync now does an auto-commit before Fetch, with the same shape as PullFrom: 
    if !s.readOnly {
    if err := s.Commit(ctx, "auto-commit before sync"); err != nil {
        if !isDoltNothingToCommit(err) {
            result.Error = fmt.Errorf("failed to commit pending changes before sync: %w", err)
            return result, result.Error
        }
    }
}
  3. internal/storage/dolt/federation_test.go:392-441 — new TestFederationSyncCommitsPendingPeerMetadataBeforeFetch. Adds a peer, asserts dolt_status is clean (catches the add-peer bug directly), then dirties federation_peers via a raw UPDATE and runs Sync against a nonexistent remote; the sync is expected to fail at fetch, but dolt_status must be clean afterward (catches the missing pre-merge commit on the Sync side). A small helper federationStatusHasTable queries dolt_status by table name.

Strengths

  • Right side of the storage-driver boundary. All edits land inside internal/storage/dolt/. Beads-side packages are untouched. This aligns with the storage-driver roadmap captured in the beads-storage-driver-roadmap memory and GH#3617: crash-safety and dirty-working-set handling are the driver's job, and that's where the fix lives.
  • Reuses existing helpers verbatim. No new commit/staging machinery is invented; the change leans on doltAddAndCommit (store.go:1715-1732) and Commit (store.go:1630-1690). Commit already correctly excludes config to avoid the GH#2455 staging-corruption pattern, so the Sync pre-commit inherits that safety.
  • Mirrors PullFrom's precedent exactly. Same readOnly guard, same isDoltNothingToCommit short-circuit, same wrapped error message shape (federation.go:60-66 vs new federation.go:294-303). This makes the change trivially auditable.
  • Honors readOnly. A read-only store will not attempt a commit; behavior degrades gracefully.
  • Targeted commit scope on add-peer. doltAddAndCommit(ctx, []string{"federation_peers"}, ...) only stages the one table, so it cannot sweep up unrelated dirty work into a "federation: add peer X" commit (GH#2455 hygiene).
  • Regression test is asserting the right property. Probing dolt_status directly is the cleanest way to detect a dirty working set without relying on the merge error message.
  • Tight diff. +69/-1, 3 files, one of them tests. Easy to review, easy to revert.

Concerns

  1. internal/storage/dolt/federation.go:294-303 — Sync's pre-commit is broader than add-peer's. Add-peer commits only federation_peers, but Sync's pre-commit calls s.Commit(ctx, "auto-commit before sync") which stages every dirty table except config. That matches PullFrom's existing behavior, but in the worst case it can produce a commit titled auto-commit before sync containing unrelated issue/dependency churn. That's pre-existing behavior in PullFrom, not a regression introduced here, and changing it is out of scope for a P0 fix. Worth a follow-up bead to consider scoping the pre-sync commit to federation/metadata tables only, or producing a richer commit message via CommitPending-style logic.
  2. internal/storage/dolt/credentials.go:300-306 — "remote already exists" string-match is unchanged but brittle. The PR did not touch this, but since AddFederationPeer's commit now follows the AddRemote call, any future variation in the remote-already-exists error string would cause the new commit to be skipped (we'd return before reaching the new code). Pre-existing risk; flag only as a future-hardening note, not a blocker.
  3. internal/storage/dolt/federation_test.go:1 — test is //go:build integration-gated. That is correct (it touches a live dolt sql-server), and matches the rest of the file, but it does mean codecov sees 0% patch coverage on the new lines. The Codecov bot comment is therefore noise, not a real signal. CI passing the integration tag remains the gate; the PR description acknowledges the local skip due to no Docker/Dolt. Not blocking.
  4. internal/storage/dolt/federation_test.go:430-441 — the helper federationStatusHasTable swallows test context via t.Helper() but is a free function rather than a method. Fine as-is; consider promoting to a shared test util if more federation tests start querying dolt_status.
  5. No CHANGELOG / docs update. FEDERATION-SETUP.md historically had a "if sync fails with stomped federation_peers, run dolt commit" workaround note (per the reporter). If that note still exists, it should be removed or updated. Worth a quick grep -ri "stomped\|federation_peers" docs/ FEDERATION-SETUP.md follow-up. Not blocking the merge.
  6. dlwiii's secondary observation about dolt gc eating issues is not addressed and shouldn't be. That's a separate issue (gc safety / docs warning) and rightly out of scope here. Filing a follow-up bead is appropriate.

Recommendations

  • Merge as-is. Outcome from PR_MAINTAINER_GUIDELINES.md: this is a clean Merge (or arguably Easy win). It solves the reported bug, follows the established PullFrom precedent, includes a targeted regression test, and is the right shape for the storage-driver-boundary roadmap.
  • File two follow-up beads (not blocking):
    1. Doc cleanup: scrub FEDERATION-SETUP.md for the obsolete dolt add -A && dolt commit -m flush workaround, since the fix makes it unnecessary.
    2. dolt gc safety: a new bead capturing dlwiii's report that running dolt gc against a bd-managed Dolt directory while bd was off appears to have eaten two issues. Whether the resolution is docs, a guard, or a driver-side fix, it shouldn't be lost.
  • On the PR thread, confirm CI ran the integration tag (so the new test actually exercised the fix), then ship it. If CI did not run integration, the maintainer-side validation should at minimum run go test -tags=integration ./internal/storage/dolt -run TestFederationSyncCommitsPendingPeerMetadataBeforeFetch -count=1 -v against a local Dolt server before merge — the PR description notes this was skipped locally.
