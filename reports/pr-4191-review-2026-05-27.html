The PR is a small, low-risk diagnostic patch by a project insider (the same person who triaged #3927). It adds opt-in debug logging on two failure paths, improves an actionable error hint, and ships focused tests. There is no behavior change for users who do not set BD_DEBUG=1 / --verbose, and the only user-visible change in normal mode is an improved error message that still satisfies the pre-existing substring assertion at internal/storage/embeddeddolt/create_issue_test.go:262.

Why it exists

Issue #3927 reports a confusing failure mode: .beads/config.yaml is present with issue-prefix: hq, but bd create fails with warning: no beads configuration found ... followed by Error: database not initialized: issue_prefix config is missing. Two things make this hard to triage:

  1. The YAML-vs-DB key-naming asymmetry is undocumented in user-facing errors: config.yaml expects issue-prefix (hyphen), the database uses issue_prefix (underscore), but the error string only shows the underscore form. A user looking at their config.yaml and the error message would not see the inconsistency.
  2. When configfile.Load(beadsDir) returns (nil, nil), the discovery path is silent — there is no indication of whether metadata.json was missing, whether config.yaml was missing, or whether something else caused the load to return empty.

This PR adds enough on-demand visibility to triage future #3927-shaped reports without shipping a new build.

What it changes

  1. cmd/bd/main.go (lines ~967-974): In the else if cfgErr == nil branch (i.e., Load returned (nil, nil)), os.Stat both metadata.json and config.yaml and debug.Logf a one-line summary so users running BD_DEBUG=1 bd create get a clear answer to "did bd find my config files at all?".
  2. internal/storage/issueops/helpers.go (ReadConfigPrefix):
    • On the missing-prefix branch, look up both issue-prefix (canonical YAML key) and issue_prefix (legacy/underscore form) from viper and debug.Logf what was seen.
    • Append "; if using config.yaml, use key 'issue-prefix', not 'issue_prefix'" to the error message so the fix is visible without BD_DEBUG.
  3. internal/storage/issueops/helpers_test.go: New TestReadConfigPrefix with two subtests using existing go-sqlmock (already at internal/storage/issueops/create_test.go:10). Covers the trimmed-prefix happy path and asserts the new error message contains the key-naming guidance.
  4. internal/config/config_test.go: New TestIssuePrefixKeyNamingInYAML that locks in viper's parser behavior — issue-prefix and issue_prefix are kept as distinct keys with distinct values — which is the assumption the new diagnostics depend on.

Strengths

  • Solves the stated problem in #3927 without behavior change. The bug class (silent failure on missing/misparsed config) gets the diagnostics needed to triage future reports, plus the most likely user error (key naming) gets a direct hint in the error itself.
  • Conservative scope. No retry logic, no new public surface, no migration. The hint is appended to the existing error string rather than replacing it; the substring "issue_prefix config is missing" still matches, so the existing assertion at internal/storage/embeddeddolt/create_issue_test.go:262 stays green.
  • Tests target the new behavior rather than just adding coverage on adjacent code. The error-message assertion uses a small containsAll helper to lock in three specific substrings — exactly the kind of test that makes a regression fail loudly.
  • Idiomatic Go and idiomatic for this codebase. debug.Logf is the project's standard mechanism (used in cmd/bd/*, internal/config, internal/molecules, internal/jira, etc.). t.Chdir and go-sqlmock are established patterns (see cmd/bd/setup_test.go, internal/storage/issueops/create_test.go).
  • No storage-driver-boundary violation. The driver-boundary memory is about not reaching into engine internals from generic packages (flocks, engine introspection, engine-specific retry/crash-recovery). This PR adds (a) a stat on two files inside cmd/bd/main.go (the orchestration layer where configfile.Load is already called) and (b) two viper lookups + a debug print inside issueops/helpers.go. internal/storage/issueops already imports internal/config (see helpers.go:17 and config_helpers.go:10); this PR introduces internal/debug to that file but internal/debug is a leaf package (no storage dependency), so no new cycle or boundary crossing.
  • Insider-authored, codex-implemented, validated locally. PR body lists the two go test invocations and git diff --check. CI patch shows all tests pass; the only "missing coverage" lines are exactly the debug.Logf branch in cmd/bd/main.go, which is unreachable without BD_DEBUG=1.

Concerns

  1. TestIssuePrefixKeyNamingInYAML (internal/config/config_test.go:1693-1722) is a viper-behavior pin, not a beads-behavior test. The test writes both issue-prefix: canonical and issue_prefix: legacy_underscore and asserts viper returns them as distinct values. That is verifying spf13/viper, not anything in this PR. It is fine to keep as a regression pin (the diagnostics in ReadConfigPrefix read both keys, so if viper ever started conflating them the diagnostics would be silently wrong), but the test name and a one-line comment should make that intent explicit. Suggested rename: TestViper_IssuePrefixKeysAreDistinct or leave the name and add // Locks in that viper treats issue-prefix and issue_prefix as separate keys, which the diagnostics in ReadConfigPrefix rely on. Minor — not a blocker.
  2. debug.Logf format string ends in \n but Logf itself does not add one. Two call sites in this PR (cmd/bd/main.go:970 and internal/storage/issueops/helpers.go:507) both pass "...\n", which matches the rest of the codebase (e.g. internal/config/config.go:300). No issue, just noting consistency.
  3. The cmd/bd/main.go diagnostic only fires on the (nil, nil) branch. Issue #3927's repro is consistent with that branch (the user sees warning: no beads configuration found), so this is the right place. However, if a future #3927-shaped report involves configfile.Load returning a non-nil cfg but with a missing/blank database name or unparsable issue-prefix, this diagnostic will not fire. Worth a follow-up bead but out of scope here: consider mirroring the stat-and-log in the cfg != nil branch when cfg.GetDoltDatabase() ends up empty / falls through to configfile.DefaultDoltDatabase. File as a separate bead rather than gating this PR.
  4. The helpers_test.go mock contains an mock.ExpectRollback() after db.Begin(), but the test uses defer tx.Rollback(). Functionally correct, but the ordering reads slightly awkward (Begin → ExpectRollback → use tx). Pre-existing pattern in this file? Not exactly — internal/storage/issueops/create_test.go uses a similar pattern. Non-blocking; reviewers used to this codebase will read past it.
  5. Patch coverage of 45% on cmd/bd/main.go is exactly the debug.Logf branch the test cannot exercise. Codecov's automated comment will look like a red flag at a glance; worth noting in the merge-button moment that the missing lines are the debug print itself (6 lines = stat calls + debug.Logf invocation), which is opt-in and cannot be covered without rigging BD_DEBUG=1 from a unit test. Not a real concern, just be ready to wave Codecov through.
