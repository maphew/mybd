Verdict

merge-fix.

The core change is correct and well-scoped: it replaces a single --reason string with an order-preserving slice flag, validates count parity, and adds focused tests. The fix matches what #3279 asked for and the embedded test reproduces the user-reported invocation pattern faithfully. CI is almost entirely green, including the differential-regression and 20-shard embedded-Dolt grids.

Two things gate the merge:

  1. CI is red on Check doc flags freshness because the contributor edited closeCmd.Long (cmd/bd/close.go:24-30) but did not regenerate docs/CLI_REFERENCE.md. The fix is mechanical — run ./scripts/generate-cli-docs.sh ./bd and commit — but it must happen before merge or the gate fails.
  2. A small but real backward-incompatibility for the --reason X --reason Y (single-ID) idiom. Previously cobra's default last-wins meant bd close id1 --reason X --reason Y closed id1 with reason "Y". New code rejects that input with got 2 close reasons for 1 issue IDs. Probably noise (no known user relied on this), but it should be called out in the PR description or release notes — or relaxed with a "treat as override when only one ID" rule.

Beyond those, suggest a small handful of polish items (most non-blocking). Lean toward landing once docs are regenerated.

Why it exists

Upstream #3279 (seanmartinsmith) reports that bd close id1 --reason='A' id2 --reason='B' id3 --reason='C' closes all three issues with reason "C", because cobra's default StringP flag is last-write-wins. The reporter explicitly notes this requires a different parsing approach (paired args or a new flag pattern). Close reasons matter for searchability and recovery context; the documented workaround was to close issues one at a time.

#3268 (slagyr) is a separate, intermittent bug where bd ready reports "No ready work found" despite open zero-dependency issues. The mybd epic mybd-r1cb flagged both for P2 cleanup. The PR description notes the author could not reproduce #3268 on current main and adds a regression guard instead of a fix.

What it changes

    • validateCloseReasons(reasons) — runs validation.ValidateCloseReason per reason (was previously single-shot).
  • New closeReasonFlagValue type (close.go:311-339) implementing pflag.Value with Set appending to a slice and Values() returning a defensive copy. Wired in by registerCloseReasonFlag(closeCmd) at close.go:293.

  • Error message format on count mismatch (close.go:364): "got %d close reasons for %d issue IDs; provide exactly one shared reason or one reason per issue".

Strengths

  • Right approach for the bug. Custom pflag.Value with Set accumulation is the standard idiom in cobra for this exact problem; using it preserves the natural shell ordering and avoids inventing a new flag-pair syntax (e.g. --reason-for id=...) that would have been worse UX.
  • Tests reproduce the actual reported invocation. The embedded test does bdClose(t, bd, dir, id1, "--reason", "A", id2, "--reason", "B") — a real spawned process exercising real cobra parsing — so a future regression in flag-slice handling would be caught.
  • Backward-compat path for shared reason is preserved. bd close id1 id2 id3 --reason X still works, and bd close --reason X id1 id2 id3 also works (verified by TestCloseResolveReasons_SharedReasonForMultipleIDs and by reading the logic — single-reason short-circuits in reasonForCloseIndex).
  • Aliases preserved as single-valued. --resolution, --message, --comment keep their existing semantics; the slice behavior is --reason-only. This is the right call: those aliases are hidden agent ergonomics, not the multi-issue path.
  • bd done <id> <reason> desire-path preserved. New code (close.go:355-358) keeps the "last positional is reason" handling for the done alias when no --reason flag was given. Also handles bd done id1 id2 reason correctly (single shared reason).
  • --reason-file interaction is sensible. A file-supplied reason becomes a single shared reason (reasons = []string{fileReason}), which is the only meaningful semantic — file body cannot be split per-ID.
  • Validation runs over every reason. When validation.on-close=error, each per-ID reason is checked; the loop in validateCloseReasons matches the per-ID semantics.
  • Refactor is a net structural improvement. Pulling the reason-derivation tangle out of Run into testable helpers makes the close command substantially easier to reason about, even apart from the bug fix.
  • CI is otherwise green. Differential regression (v0.49.6 baseline), 5 cross-version upgrade smokes (v1.0.0–v1.0.4), all 20 embedded-Dolt shards, macOS, Windows smoke, Nix flake, storage-domain — all pass.

Concerns

  1. (blocker) Check doc flags freshness is failing because docs/CLI_REFERENCE.md line ~346 does not include the two new lines added to closeCmd.Long ("When closing multiple issues, provide one --reason for all IDs or repeat --reason once per ID in the same order."). The CI log explicitly says Run: ./scripts/generate-cli-docs.sh ./bd. Fix is one-command-and-commit; must happen before merge or this required gate stays red.
  2. (minor backward-compat break, not a blocker) The previous --reason was a normal cobra string with last-wins. bd close id1 --reason X --reason Y used to close id1 with reason "Y"; new code returns got 2 close reasons for 1 issue IDs and exits non-zero. Probably no one was doing this intentionally, but a shell-history rerun or an alias that double-supplies --reason will now fail. Two cheap options:
    • (a) Mention this in the PR description and release notes.
    • (b) Relax: when len(args) == 1, accept any number of --reason flags and use the last (preserving old behavior for the one case that's unambiguous). Suggested edit at close.go:363: 
      if len(reasons) > 1 && len(args) == 1 {
    reasons = reasons[len(reasons)-1:]
}
if len(reasons) > 1 && len(reasons) != len(args) { ... }
    Either is fine; option (b) is more conservative and costs three lines.
  3. (nit) Error message at close.go:364 reads "got 2 close reasons for 3 issue IDs; ...". When len(args)==1 you would see "got 2 close reasons for 1 issue IDs" — the trailing IDs is plural in a singular case. Trivial polish: "got %d close reasons for %d issue ID(s)" or branch on len(args)==1.
  4. (nit, won't bite this PR but worth noting) closeReasonFlagValue is registered against the package-global closeCmd via init(). The same *closeReasonFlagValue pointer persists for the life of the process. The CLI runs one-shot per invocation so this is fine, but if anyone starts driving closeCmd.Execute() in-process (the embedded-Dolt tests do not — they spawn subprocesses, confirmed at close_embedded_test.go:24-32), reasons will accumulate across invocations. A Reset() method or a fresh-value-per-Run construction would future-proof it. Suggest adding a one-line cobra.OnInitialize-style reset, or accepting it and adding a // CAUTION: not safe for in-process Execute reuse comment near close.go:311.
  5. (nit) Patch coverage 37.87% (Codecov) — 36 missing + 5 partial in close.go. Most of the new error-handling branches are unexercised, particularly the collectCloseReasonFlags cmd.Flags().GetString(name) error arm (which can only fire if the flag was registered with the wrong type, i.e. dead code in practice). The slice-flag Type() and String() methods are also uncovered. Worth a single unit test that walks each alias (--resolution, --message, --comment) and asserts each produces a []string{value} from collectCloseReasonFlags. Not a blocker; the integration coverage of the happy paths is solid.
  6. (nit) Values() []string returns a defensive copy (close.go:335-339) — good. But the duck-typed interface assertion at close.go:371 (if v, ok := flag.Value.(interface{ Values() []string }); ok) means if someone replaces closeReasonFlagValue with another pflag.Value implementation that doesn't expose Values(), the function silently falls through to the alias scan and the multi-reason flag stops working without any warning. A type-asserted access (flag.Value.(*closeReasonFlagValue)) would fail loudly. Style preference; current pattern is more decoupled.
  7. (nit) --suggest-next (close.go:74-76) and --continue (close.go:69-71) already enforce single-ID. Worth a sentence in the help text saying "with multiple IDs, per-issue --reason flags must match positional order" or pointing at the Long description — the current --reason help string is just "Reason for closing", which doesn't hint at the per-ID syntax. Cheap polish, low priority.
  8. (observation) The #3268 regression guard (ready_includes_open_issue_with_zero_dependencies) only proves current main produces ready output for a zero-dep issue. The original bug report (slagyr) describes an intermittent failure under status.custom = unverified:wip + workflow.done-status = unverified — neither of which the new test exercises. The author's PR description is honest about this ("current main did not reproduce... this adds a regression guard"), so it is not a misrepresentation. But anyone scanning the PR title may assume #3268 is fixed; the title says only fix(close), which is accurate. No action needed unless you want to clarify in the merge commit that #3268 remains open.
  9. (observation) PR branch is based on 4e387914a (2026-05-26). Upstream main has since merged 7fc2374ed Merge pull request #3533 (dolt mode config), be0aa7072 (init server-mode), and a650497ae Merge pull request #4185 from coffeegoddd/db/provider (a 4k-line refactor that deletes a lot of the deprecated cmd/bd/create_proxied_* and internal/storage/domain/db/issue.go). Backwards compatibility & migration

  • Compatible:
    • bd close id1 (no reason) — unchanged.
    • bd close id1 --reason X — unchanged.
    • bd close id1 id2 id3 --reason X — unchanged (shared X applies to all).
    • bd close --reason X id1 id2 id3 — unchanged (shared X applies to all).
    • bd close id1 --resolution X / --message X / --comment X — unchanged.
    • bd close id1 --reason-file=path — unchanged.
    • bd done id1 reason — unchanged (desire-path preserved).
    • bd done id1 id2 reason — unchanged (single shared reason).
  • Newly working (the actual fix):
    • bd close id1 --reason A id2 --reason B id3 --reason C — now closes each with its own reason instead of all with "C".
  • Newly rejected (the minor break, see Concern 2):
    • bd close id1 --reason X --reason Y (single ID, repeated flag) — used to take "Y" (cobra last-wins), now errors got 2 close reasons for 1 issue IDs.
    • bd close id1 id2 --reason X --reason Y --reason Z (any count mismatch where reasons > 1) — used to take "Z" for both, now errors.
  • JSON output: shape unchanged. closedIssues is built per-ID and the JSON envelope (map[string]interface{}{"closed": ..., "unblocked"/"continue"/"claimed": ...}) is identical. Per-ID reasons appear in each issue's stored CloseReason field, which is what users would inspect via bd show --json or in scripts.
  • Audit log: audit.LogFieldChange(id, "status", oldStatus, "closed", actor, reason) now records the per-ID reason — strictly an improvement.
  • Release-note line suggestion: "Closing multiple issues with bd close A --reason X B --reason Y now records each reason against its issue. Supplying two or more --reason flags whose count does not match the positional ID count (and is not exactly 1) now exits with a clear error rather than silently using the last value."
