Verdict

merge (with two small follow-ups, neither blocking).

The PR does exactly what its title says: it removes the three concrete obstacles to running uv run pytest + uv run python -m build cleanly inside integrations/beads-mcp/, and it does so without expanding the contract beyond what is achievable today. Hard-coded /Users/stevey/... paths are replaced with PROJECT_ROOT = Path(__file__).resolve().parents[1], the bd_client integration fixture moves from a shared /tmp/beads_test_workspace_* + collision-prone /tmp/embeddeddolt/test layout to a per-test tmp_path / "workspace" with explicit BEADS_DIR, and build>=1.3.0 is added to the dev dependency group so python -m build is reproducibly available. The README explicitly demotes mypy/ruff from "baseline" to "source maintenance, not part of baseline validation yet" — an honest documentation move rather than a hidden capability loss. CI is unaffected because (as confirmed by reading .github/workflows/) no upstream workflow actually runs uv run pytest on this directory; the baseline is a developer/maintainer contract, not a gated check.

The work matches the linked beads mybd-hbyb plan (worker delegation, integration-only, no source changes), and the upstream change is bounded and reversible.

Why it exists

While reviewing dependabot PR gastownhall/beads#3467 (a pydantic bump) on top of the merged #3466, the documented MCP validation sequence (uv sync && uv run pytest && uv run mypy && uv run ruff && python -m build) failed at multiple stages on a refreshed PR head — not because of the dependency bump, but because the baseline was rotten. The concrete failures, per bd show mybd-hbyb:

  1. uv run pytest: tests/test_compaction_config.py hard-coded /Users/stevey/src/dave/beads/integrations/beads-mcp as the subprocess cwd and as a file-read path, so the test silently produced empty stdout on any non-stevey machine. The bd_client integration fixture seeded /tmp/beads_test_workspace_* and let bd create /tmp/embeddeddolt/test, which collided with any prior test run.
  2. uv run mypy src/beads_mcp: pre-existing type errors in bd_client.py and server.py.
  3. uv run ruff check src/beads_mcp: pre-existing lint/format findings in source.
  4. python -m build: the build module was not in the dev group, so the command failed with No module named build.

Cumulative effect: any dependabot or contributor PR review on integrations/beads-mcp/ was forced to triage noise from the broken baseline before anything else. This PR's job is to stop that noise.

What it changes

integrations/beads-mcp/README.md (lines 248-272 of new content):

  • Adds a top-level "Baseline validation" code block that documents the canonical sequence as uv sync → uv run pytest → uv run python -m build.
  • Re-titles the existing mypy and ruff sections with the parenthetical "source maintenance, not part of baseline validation yet". The commands are preserved verbatim — only the contract changes.

integrations/beads-mcp/pyproject.toml (one-line [dependency-groups].dev insertion at line 94):

  • Adds "build>=1.3.0". uv.lock is updated in lockstep with the new transitive deps (pyproject-hooks, conditional colorama, tomli, importlib-metadata).

integrations/beads-mcp/tests/test_compaction_config.py (whole-file changes, +44/-26):

  • Adds PROJECT_ROOT = Path(__file__).resolve().parents[1] at module load (line 12).
  • Adds a _python_env() helper (lines 14-21) that prepends PROJECT_ROOT/src to PYTHONPATH so subprocesses can import beads_mcp without depending on cwd. This is a real upgrade: previously the subprocess only worked because cwd=/Users/stevey/... happened to also be the project root that contained src/.
  • Replaces both cwd="/Users/stevey/.../beads-mcp" literals with cwd=PROJECT_ROOT (lines 53, 79).
  • Replaces both open("/Users/stevey/.../server.py") / open(".../CONTEXT_ENGINEERING.md") literals with open(PROJECT_ROOT / "src" / "beads_mcp" / "server.py") and open(PROJECT_ROOT / "CONTEXT_ENGINEERING.md") (lines 204, 213).
  • A large number of formatting/whitespace fixes (trailing whitespace on blank lines, blank line after import pytest inside try blocks) that bring the file into ruff format compliance.

integrations/beads-mcp/tests/test_bd_client_integration.py (largest single change, +56/-60):

  • Removes import tempfile.
  • Replaces the module-shared temp_db fixture (which used tempfile.mkstemp(..., dir="/tmp") and a single .db file) with a temp_workspace(tmp_path) fixture (lines 36-40) that returns a per-test tmp_path / "workspace" directory. tmp_path is the pytest built-in — it gives each test a unique directory under pytest.gettempdir()/pytest-<user>/test-NNN/, which is naturally collision-free.
  • Rewrites the async bd_client fixture (lines 49-77):
    • Constructs BdClient(bd_path=..., beads_dir=str(beads_dir), working_dir=str(temp_workspace)) instead of the deprecated beads_db parameter.
    • Exports BEADS_DIR=<temp>/workspace/.beads and BD_NON_INTERACTIVE=1, and pops any inherited BEADS_DB.
    • Calls bd init --prefix test --non-interactive --skip-agents --skip-hooks. The new flags exist in cmd/bd/init.go:1560-1573 — confirmed: --skip-hooks, --skip-agents, --non-interactive are all real.
    • Drops the nested with tempfile.TemporaryDirectory(...) wrapper because the tmp_path fixture already provides cleanup.
  • Rewrites test_init_creates_beads_directory (lines 419-449):
    • Switches from tempfile.TemporaryDirectory(prefix="beads_init_test_", dir="/tmp") to tmp_path / "init-workspace".
    • Uses monkeypatch to set/clear BD_NON_INTERACTIVE, BEADS_DIR, and BEADS_DB — actual env-restore guarantees.
    • Changes the post-init assertions from "a *.db file named beads.db exists" to the embedded-Dolt reality: metadata.json, config.yaml, and embeddeddolt/test/.dolt/ directory. This is the correct shape for the current backend — the old assertion would have falsely passed only on the SQLite-era backend.
  • Reformats one line for ruff (test_claim_issue line 159, single-line CreateIssueParams call).

integrations/beads-mcp/uv.lock (+27 lines):

  • Lockfile delta for build==1.5.0 plus its transitive deps (pyproject-hooks==1.2.0, conditional colorama, tomli, importlib-metadata).

Net gate change: pytest is fixed, python -m build is fixed, mypy and ruff src/ are formally moved out of the baseline (they remain present in the dev group, just demoted in docs).

Strengths

  • Targeted, no scope creep. Exactly the four failure modes called out in mybd-hbyb are addressed; no incidental source refactors.
  • Per-test isolation is real. tmp_path is the right primitive — pytest 9 creates per-test directories under a session base, so two pytest -n auto workers will not collide, and prior runs do not pollute. The fragile /tmp/embeddeddolt/test collision is gone.
  • The init test is now more accurate, not just hermetic. Asserting metadata.json + config.yaml + embeddeddolt/test/.dolt/ matches what bd init actually creates with the embedded-Dolt backend; the old beads.db assertion was a holdover from the SQLite backend and would have started passing under false pretenses.
  • PYTHONPATH prepend pattern is robust. _python_env() adds PROJECT_ROOT/src so the subprocess can import beads_mcp even if uv run is invoked from elsewhere. This is independent of cwd and therefore more robust than the old "hope cwd lines up" approach.
  • Documentation honestly tracks reality. Demoting mypy/ruff from "baseline" to "source maintenance, not part of baseline validation yet" is preferable to silently dropping them. The "yet" leaves a clear hook for a follow-up to re-promote once the source side is cleaned.
  • PR description follows project convention (Why / What / Validation / Bead / signed). Commit is single, focused, with test(mcp): prefix.
  • CI green across all 44 checks including the storage-domain and Embedded-Dolt suites; Codecov reports patch coverage on all modified lines.

Concerns

  1. (nit) mypy and ruff demoted in docs but never re-promoted; no tracking issue is linked. integrations/beads-mcp/README.md line 261 says "source maintenance, not part of baseline validation yet" but the README does not reference a follow-up beads issue or PR for the source cleanup. Future readers will not know what "yet" means. Suggested action: append (tracked by <bead>) or similar, and file a follow-up bead for the actual mypy --strict and ruff check src/beads_mcp cleanups so this demotion is a checkpoint and not an indefinite waiver. Not a blocker — the existing baseline is honest.
  2. (nit) conftest.py safety check still guards BEADS_DB and BEADS_WORKING_DIR, not BEADS_DIR. integrations/beads-mcp/tests/conftest.py lines 28-58 protect against accidentally pointing tests at the production .beads/ via BEADS_DB or BEADS_WORKING_DIR. The new bd_client fixture exports BEADS_DIR=<tmp>/.beads, which is the new primary handle in the BdClient (bd_client.py:302-303). If a future contributor sets BEADS_DIR=/path/to/production/.beads in their shell, the conftest will not catch it. Not introduced by this PR, but this PR is the natural place to extend the guard. Suggested action (follow-up, not blocking): mirror the existing production_beads check for BEADS_DIR in conftest.py:pytest_configure.
  3. (nit) test_compaction_config.py still mutates global os.environ directly instead of using monkeypatch. The PR fixes the path hard-coding and the formatting but leaves the os.environ["BEADS_MCP_COMPACTION_THRESHOLD"] = "..." + try/finally pattern (e.g. lines 107-114, 124-131, 137-143, 155-159, 167-173) untouched. Compare with the new test_init_creates_beads_directory in test_bd_client_integration.py which adopts monkeypatch cleanly. A pytest.fixture(autouse=True) that snapshots/restores the relevant env vars, or just per-test monkeypatch.setenv, would be more robust against test-run interruption (Ctrl-C between try and finally leaves the process env dirty for any caller-driven subsequent uses). Not a regression — this was the prior style — but the file is being touched anyway, so this would have been a cheap win.
  4. (nit) The bd_client fixture leaks one line that ruff/lint would flag if mypy --strict were re-enabled. test_bd_client_integration.py:55 builds env and exports BEADS_DIR, but the BdClient itself is constructed with the same beads_dir two lines earlier, so the env is only consumed by the init subprocess. That is correct (init needs env, follow-up calls re-derive env from BdClient). Just worth a one-line comment to that effect, because the dual mechanism (constructor arg + env var) reads as redundant on a quick scan.
  5. (nit) bd init --non-interactive --skip-agents --skip-hooks is a CLI surface this test now hard-binds to. Confirmed in cmd/bd/init.go:1560-1573 that those three flags exist on main, so the test is correct today. But if any of those flags are renamed or removed (--skip-agents is comparatively young), the test will start failing with a fixture-setup error rather than a meaningful test failure. Worth a one-line pytest.fail guard that runs bd init --help once at session scope and asserts the flags are present, so the failure message points at the cause. Cosmetic; do as a follow-up.
  6. (nit) Test pollution detected (bd-2c5a) exit path is now unreachable from the integration test because BEADS_DB is popped early. This is intended — the fixture explicitly does env.pop("BEADS_DB", None) (line 56) — but the conftest's runtime safety check (pytest_runtest_setup, line 65) keys off BEADS_DB. With this PR, that runtime guard never fires for the bd_client fixture. The guard still protects accidental production-pointing on other fixtures, so this is not a regression, just a note that the guard's coverage shrank a little.

None of the above are blockers. The PR can land as-is.

Storage-driver-boundary check

No violations.

The PR touches Python integration tests and one Python dependency entry. It does not import or call any Go storage driver, does not assert on any storage internal beyond observing that embeddeddolt/test/.dolt/ exists after bd init (which is a black-box file-existence check on the result of a public CLI command, not a reach across the driver boundary). The previous /tmp/embeddeddolt/test collision risk is actually eliminated by this PR — that earlier shared path was, in spirit, a leak of storage layout into the test fixture. The new fixture lets bd put the embedded-Dolt directory wherever it wants under tmp_path / "workspace" / ".beads", with no test code depending on the layout except the one positive existence assertion in test_init_creates_beads_directory. That assertion is appropriate because the test's whole point is "init created the directory I asked it to".

Backwards compatibility & migration

  • Validation command for contributors changes. The documented baseline drops uv run mypy and uv run ruff check src/beads_mcp and adds uv run python -m build. Any contributor checklist or PR template that says "run mypy and ruff" needs updating. The README change captures this; no external doc was found in bd-main/ that hard-codes the old sequence.
  • CI is unaffected. Searched .github/workflows/ — only release.yml and test-pypi.yml touch integrations/beads-mcp, and they only run uv build. Nothing runs pytest, mypy, or ruff on this directory upstream, so this PR cannot break CI.
  • Test infrastructure: the bd_client fixture's signature is internal; no test outside this file depends on it. The replacement of temp_db with temp_workspace is invisible to test bodies, which only reference bd_client.
  • Python dependency surface grows by one direct dep. build>=1.3.0 is a well-maintained PyPA project with a small transitive footprint (packaging, pyproject-hooks, plus conditional colorama/tomli/importlib-metadata on older Pythons). Acceptable.
  • No source-side migration needed. Nothing in src/beads_mcp/ is touched.
