Beads Codebase Health Assessment

Date: 2026-04-24 · Scope: local checkout ./bd-main, upstream gastownhall/beads

Executive summary: Beads is in a workable but stressed state. It has real engineering assets: a strong test culture, broad CI, regression and migration gates, meaningful documentation, and a clear product thesis. It is also carrying the normal scars of a fast, model-heavy first year: a very large CLI package, transitional global state, broad interfaces, duplicated storage behavior across server and embedded Dolt modes, release and packaging complexity, and a backlog clustered around data safety.

Overall health: C+ / B-

The codebase is shippable with discipline, but refactoring should now become a first-class program rather than opportunistic cleanup. The safest path is staged extraction, not a rewrite.

What Is Healthy

Test Investment Is Real

The repository has a high ratio of tests to implementation. Large suites exist around CLI behavior, Dolt storage, migrations, issue operations, tracker integrations, regression scenarios, and embedded Dolt flows. CI also separates fast PR checks, embedded Dolt checks, nightly full tests, regression tests, migration tests, release checks, documentation deployment, and Nix builds.

Product Boundaries Are Visible

The domain is visible through internal/types, internal/storage, internal/storage/dolt, internal/storage/embeddeddolt, internal/storage/issueops, external tracker packages, UI helpers, validation, config, and Dolt server packages. The architecture is not absent. It is partially submerged under CLI orchestration.

Release Engineering Is Serious

The project has GoReleaser, npm packaging, PyPI integration for MCP, Docusaurus docs, winget metadata, Nix support, install scripts, checksum/security docs, and cross-version smoke tests. That indicates the project has crossed into product territory.

The Backlog Names the Right Problems

The local issue tracker already clusters risk around embedded/server Dolt reliability, remote/sync data safety, admin/status command regressions, ready/list JSON behavior, and packaging portability. That aligns with the code evidence.

Primary Risks

1. cmd/bd Is Too Large

cmd/bd has 613 Go files and roughly 183k lines. It imports most internal packages directly and still contains global state, command lifecycle logic, storage selection, formatting, error exits, hooks, telemetry, profiling, and domain workflows.

Refactoring direction: keep Cobra as the adapter, move command behavior into injectable services, and make process exit and terminal output adapter concerns.

2. Storage Interfaces Are Too Broad

Storage and DoltStorage expose issue CRUD, dependencies, labels, queries, wisps, events, config, transactions, merge slots, version control, sync, federation, compaction, and advanced queries. That breadth makes every concrete store feel responsible for the whole product.

Refactoring direction: introduce narrow ports such as IssueReader, IssueWriter, DependencyStore, VersionedStore, RemoteSyncStore, and MaintenanceStore.

3. Dolt Lifecycle Is the Highest Operational Risk

The most concerning live issues are around embedded startup, lock lifecycle, journal corruption, bootstrap stale server state, watcher races, migration idempotence, server auto-start, stale locks, and remote push/pull safety.

Refactoring direction: treat Dolt lifecycle as its own subsystem with a small state machine and scenario tests.

4. Process Exit and JSON Contracts Need Hardening

Direct os.Exit calls in command code make failures harder to test and make JSON/error contracts harder to preserve. Commands should return typed results and typed errors, then a single CLI runner should map them to text, JSON, and exit codes.

5. Documentation Is Extensive but Likely Drifty

The docs are strong, but there are many overlapping surfaces: root docs, versioned site docs, integration docs, agent docs, generated references, setup docs, testing docs, and old design notes. Canonical repository URLs and install instructions should be normalized.

Recommended Refactoring Program

Phase 0: Stabilize Before Large Refactors

Resolve the bd-main Dolt history divergence, land or reject high-priority embedded/server lifecycle fixes, make auto-push policy predictable, and keep the short Go suite green.

Phase 1: Create a CLI Service Layer

Introduce services for high-churn commands such as ready, list, show, create, update, close, dolt, and doctor. Cobra should parse and render, not own behavior.

Phase 2: Narrow Storage Capabilities

Add narrow interfaces at use sites, move common SQL issue behavior into issueops, and add contract tests that run against both server and embedded implementations where behavior should match.

Phase 3: Make Dolt Lifecycle a State Machine

Centralize state detection for uninitialized, embedded, server, unavailable, migration-required, lock-contended, remote-configured, remote-diverged, and recovery-required states.

Phase 4: Clean Packaging and Docs Drift

Normalize repository URLs and install paths across README, package metadata, PyPI metadata, website, and scripts. Add release-blocking consistency checks.

GPT-5.5 Refactoring Rules

  1. Start from a Beads issue with a single behavioral objective.
  2. Capture existing behavior with a failing or characterization test.
  3. Extract one boundary at a time.
  4. Preserve CLI output and JSON contracts unless the issue explicitly changes them.
  5. Run targeted tests first, then the short suite.
  6. File follow-up issues instead of expanding the patch.

Bottom Line

Beads is past the point where "vibe-coded" is the right operating model. It has enough users, storage complexity, tests, packaging, and recovery surface that it needs product-grade maintenance. The good news is that the foundations are much better than a typical one-year prototype: the tests and CI are strong enough to support careful refactoring.

The right next move is a stabilization and extraction campaign, not a rewrite.
