beads v1.1.0 Release Readiness — 2026-07-01

Scan of upstream gastownhall/beads, the v1.1.0-rc.1 release candidate, docs, packaging channels, and the local verify queue. Verdict up front:

The RC is NOT ready to promote to stable. Two open migration regressions are filed directly against v1.1.0-rc.1, the release stability gate has not been run, and three release-relevant fixes landed on main after the RC tag. Plan for an rc.2 from current main (plus fixes below) rather than promoting rc.1.

1. Release blockers (code)

#4502 — v53 migration fails on v52 DBs (OPEN, unfixed on main)

0053_repair_rig_wisps.up.sql INSERTs rig/agent columns (hook_bead, role_bead, agent_state, last_activity, role_type, rig) into issues, but older standalone DBs can be at schema_migrations v52 with an issues table that predates those columns. The migration's existing guard only checks that the wisps table exists — it does not check the issues columns — so the failure reproduces even with zero rig wisps, and it is still present on current main (verified by reading internal/storage/schema/migrations/0053_repair_rig_wisps.up.sql at upstream/main 9ee97c264).

This is exactly the N-2 embedded "verify schema migration" row of the release stability gate matrix (docs/RELEASE-STABILITY-GATE.md), which has a no-override rule: gate failures block the release.

Fix shape: extend the INFORMATION_SCHEMA guard to also require the expected issues columns (or repair/add them first), mirroring the pattern already used for wisp_dependencies.id.

#4534 — bd create bricked after v53 migration (OPEN, filed 2026-07-01)

On 1.1.0-rc.1 (fa4dce454), embedded Dolt: every bd create — including brand-new top-level issues with no --parent — fails with fk_counter_parent FK violation on child_counters. Reads fine, bd doctor clean, started immediately after a "schema: apply migrations" commit. Looks like an insert-ordering / FK-visibility regression on the write path post-migration. Untriaged (no labels yet). Must be root-caused before a stable cut: a release that can brick issue creation on an otherwise healthy DB is a repeat of the v1.0.5 gating incident.

3. Upgrade documentation

Good news: the canonical upgrade recipe exists and is consistent.

  • website/docs/getting-started/upgrading.md is canonical: per-method upgrade table, canonical Homebrew tap-migration block, single-clone and multi-clone BD_ALLOW_REMOTE_MIGRATE=1 bd migrate recipes (#4514, validated end-to-end on a live v49→v53 DB).
  • docs/INSTALLING.md "Updating bd" mirrors it exactly and links out; the tap-migration snippet is byte-identical across all three copies.
  • docs/RELEASING.md is a pointer stub to root RELEASING.md — no drift.
  • No stale hardcoded versions in install.ps1, scripts/install.sh, npm-package (at 1.1.0-rc.1, bumps at promotion), or install docs.

Gaps to close before the final:

  1. No backup step in the migration recipes. Both the single-clone and multi-clone recipes go straight to BD_ALLOW_REMOTE_MIGRATE=1 bd migrate with no prior bd export / Dolt-level backup. The #4502 reporter's own first move was a JSONL export — the docs should mandate it. Add an explicit pre-migrate backup step to upgrading.md (and the CHANGELOG Upgrade Notes recipe).
  2. README.md has no upgrade section or pointer — zero hits for "upgrade". A front-page reader never learns the migration gate exists. One short paragraph + link to the upgrading guide is enough.
  3. #4502 failure class is documented nowhere (no doc mentions schema-version-vs-actual-columns drift). Once the fix ships, the CHANGELOG entry should carry an upgrade note; until then it belongs in Known Issues on the rc release page.
  4. CHANGELOG: [Unreleased] is empty — #4514/#4515/#4516 are not logged anywhere. A [1.1.0] final entry must be written (RELEASING.md requires it, plus a cmd/bd/info.go 1.1.0 whats-new entry at promotion).
