## Summary

A minimal, correct fix to `TestGeminiSettingsPaths` which was failing on Windows because the test compared hardcoded forward-slash literals against the output of `filepath.Join`, which returns OS-native separators. The change mirrors an identical pattern already established in sibling tests (`TestMuxProjectAgentsPath`, `TestMuxGlobalAgentsPath` in `mux_test.go`). No production code changes — this is purely a test-side repair.

## Concerns

**Cross-platform correctness (central question)**: The fix is correct in intent but has a subtle residual issue. The test still passes Unix-rooted literals as the base argument:

```go
geminiProjectSettingsPath("/my/project")   // gemini_test.go:470
geminiGlobalSettingsPath("/home/user")     // gemini_test.go:475
```

On Windows, `/my/project` is a relative path (no drive letter), so `filepath.Join("/my/project", ".gemini", "settings.json")` yields `\my\project\.gemini\settings.json` — a relative path rooted to the current drive. The test now passes on Windows in the sense that both sides of the comparison use `filepath.Join` and will produce the same string, so the assertion holds. However, the test is not exercising realistic Windows paths (e.g., `C:\Users\user`). This is a pre-existing limitation, not introduced by this PR, but it's worth noting: the fix eliminates the Windows CI failure without fully validating Windows path semantics.
