# PR Review: #3267 — fix(create): add --rig flag to bd create for Gas Town rig routing

**Author:** rohanmyer
**Verdict:** approve-with-nits

---

## Summary

`gt done` (the Gas Town CLI for finishing polecat work) calls `bd create --rig=<name>` to route a new MR bead into a specific rig's beads database, but `bd create` had no `--rig` flag, causing a hard failure and a stalled merge queue. This PR adds that flag, resolving the rig name to a filesystem path via `$GT_ROOT`: `hq` maps to the town root, any other name maps to `$GT_ROOT/<rig>/mayor/rig`. The change is small (29 additions, 1 deletion, 1 file) and directly unblocks a real operational breakage.

---

## Strengths

- **Tight scope:** One file, one concern. The `resolveRigName` helper is cleanly isolated and easy to test independently.
- **Precedence ordering is correct:** `--rig` beats `--repo` beats auto-routing, matching the stated specificity intent.
- **Error messages are actionable:** The `GT_ROOT is not set` message includes recovery advice ("run inside a Gas Town session").
- **Fallback env var documented:** `GT_TOWN_ROOT` fallback is present and mirrors the pattern used elsewhere in the codebase (`formula.go`, `dolt.go`).
- **No behavior regression:** The `else if` chain means existing `--repo` and auto-routing paths are untouched.

---

## Concerns

**1. Commit type mismatch (`fix` vs `feat`)** — Adding a previously absent flag is a feature, not a fix. Using `fix` obscures this in changelogs and semantic-version tooling. This should be `feat(create): add --rig flag for Gas Town rig routing` with a note in the body that it unblocks a `gt done` regression.

**2. Path is assumed, not validated (`cmd/bd/create.go:357` in PR)** — `resolveRigName` returns `$GT_ROOT/<rig>/mayor/rig` without checking whether that directory (or its `.beads/` subdirectory) actually exists. The first user-visible failure will be an opaque Dolt error from `newDoltStoreFromConfig`, not a friendly "rig 'mycp' not found at /…/mycp/mayor/rig". Upstream `ensureBeadsDirForPath` is called later but expects a valid parent, so a typo in the rig name will produce a confusing error. A `stat` check in `resolveRigName` with a clear message would be a significant usability improvement.

**3. No test coverage** — The `resolveRigName` function is trivially unit-testable (pure env + string logic) yet has no tests. Given the stated real-world failures across multiple polecats, a table-driven test covering `hq`, a normal rig name, an empty string, and missing `GT_ROOT` would prevent regressions and serve as documentation.

**4. `gastown` is not a special case** — The PR description's test example `bd create --rig=gastown` routes to `$GT_ROOT/gastown/mayor/rig`, yet the PR body describes `hq` as the town-level alias. If `gastown` is also a town-level alias in practice, the mapping logic is silently wrong; if it isn't, the test example is misleading and should be removed or clarified.

---

## Suggestions

- `cmd/bd/create.go` (new `resolveRigName`): Add an `os.Stat` check on the returned path and return a descriptive error if it does not exist, e.g. `"rig %q not found at %s — check rig name or GT_ROOT"`.
- `cmd/bd/create.go` (new `resolveRigName`): Add a `_test.go` unit test file (or extend `create_test.go`) with `TestResolveRigName` covering the four cases above.
- Commit subject: change `fix(create):` to `feat(create):` per Conventional Commits spec.

---

## Test Coverage

The PR body includes three manual `--dry-run` invocations as a smoke test, which is better than nothing. There are zero automated tests for the new code path. Given that the existing `cmd/bd/` test files are extensive, the absence of a unit test for `resolveRigName` is a gap worth addressing before merge.
