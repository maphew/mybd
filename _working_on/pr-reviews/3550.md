The fix is correct, narrowly scoped, and well-reasoned. The extracted predicate is pure and fully tested. One real concern (see below) is that `IsAutoStartDisabled()` reads ambient env/config at call time from inside the command's `Run` closure, making the routing decision implicitly depend on global state rather than the already-loaded `cfg` â€” a minor testability and coupling issue.

---

## Summary

The externally-managed case arises when a rig is configured as `dolt_mode: server` pointing at `localhost` but `dolt.auto-start: false` (e.g., an orchestrator or systemd owns the server lifecycle). Because bd never wrote a PID file for that server, the old PID-file path returned `Running: false`, lying to agents. The fix widens the existing SQL-probe path â€” already used for non-local hosts â€” to also cover local hosts where auto-start is disabled, since in both cases bd has no PID file and the only reliable signal is a live SQL ping.

---

## Strengths

- **Minimal blast radius.** Only `doltStatusCmd` is touched; the PID-file path for bd-managed (auto-start enabled) servers is completely unchanged.
- **Pure predicate.** `shouldUseExternalDoltStatus` takes `autoStartDisabled bool` as a parameter rather than reading env/config itself, making it straightforward to unit-test without environment manipulation.
- **Thorough table-driven tests.** Eight cases in `TestShouldUseExternalDoltStatus` cover nil cfg, embedded mode, remote host, local host with both auto-start states, empty host, and `localhost` literal. Env vars are cleared with `t.Setenv` to prevent ambient contamination.
- **Comment quality.** The GoDoc on the new function clearly articulates the two routing branches and the fallback semantics.
- **Existing SQL-probe timeout is already present.** `runExternalDoltStatus` already uses a 5-second `context.WithTimeout` and returns a structured error on unreachable servers, so the newly routed path inherits correct timeout/error handling for free.
- **JSON output fidelity.** The PR body confirms the JSON shape changes from `{"running": false, "pid": 0, ...}` to `{"running": true, "mode": "external", ...}`, which is the truthful representation.

## Concerns

### 1. `IsAutoStartDisabled()` is called outside `cfg` â€” implicit global dependency (main concern)

`cmd/bd/dolt.go` line ~468 (post-patch):
```go
if cfg, cfgErr := configfile.Load(beadsDir); cfgErr == nil && cfg != nil &&
    shouldUseExternalDoltStatus(cfg, doltserver.IsAutoStartDisabled()) {
```

`IsAutoStartDisabled()` reads `os.Getenv("BEADS_DOLT_AUTO_START")` and the global `config.GetString("dolt.auto-start")` at invocation time. The `cfg` passed to `shouldUseExternalDoltStatus` was loaded from a specific `beadsDir`, but `IsAutoStartDisabled()` reads the *active* (globally-bound) config, which may differ in multi-worktree or context-switched scenarios. The predicate itself is pure, but its caller is not. This is a pre-existing pattern in the codebase, not introduced here, but the PR would be cleaner if `cfg.AutoStart` (if that field exists) were used instead of the global helper â€” or at minimum a comment noting the intentional coupling.

### 2. `configfile.Load` error is silently swallowed

Lines ~466-469: if `cfgErr != nil`, the condition short-circuits and falls through to the PID-file path without any warning. For the new auto-start-disabled case this means a corrupted or missing config file produces a silent false negative (reports "not running" instead of "error reading config"). The remote-host case had the same issue before, so again this is not newly introduced, but the expanded coverage makes it more likely to surface.

### 3. No integration/smoke test for the locally-managed path preservation

The test plan explicitly marks unchecked: `bd dolt status` against a bd-managed local server still prints PID/log/data. The CI gate only covers the predicate unit test and the already-existing `TestRunExternalDoltStatus_Unreachable`. A test that mocks `doltserver.IsRunning` returning a running state and asserts the PID/log output is not added, leaving the "preserved path" claim unverified by automation.

### 4. Output format change for automation consumers (low severity)

The PR body notes the JSON shape change. This is the correct fix, but any automation that parses `bd dolt status --json` and checks for `"running": false` as a sentinel for "needs restart" will now get `"running": true` on affected rigs. This is a **breaking change in observable behavior**, even though it's the right behavior. A CHANGELOG entry or migration note would help operators.

---

## Suggestions

- **`cmd/bd/dolt.go` ~468**: Consider a comment like `// IsAutoStartDisabled reads global config/env; consistent with other callers in this file` to document the intentional coupling rather than leaving it implicit.
- **`cmd/bd/dolt.go` ~466-469**: Add a `cfgErr != nil` branch that logs a warning (to stderr) before falling through, matching the error-reporting pattern used elsewhere in `doltStatusCmd`.
- **`cmd/bd/dolt_test.go`**: Add a table entry or separate test that calls the full `Run` closure with a fake `beadsDir` where `IsRunning` would return a live PID state and asserts the bd-managed output path is still taken, closing the gap in the test plan.
- **CHANGELOG.md / release notes**: Note the JSON output shape change for `bd dolt status` on externally-managed local servers so operators can update any automation that parses it.
