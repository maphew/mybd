Verdict: request fixes locally before landing.

The proxy package refactor itself looks directionally sound, and targeted proxy tests passed locally. The PR is not merge-ready because the advertised bd init --proxied-server path is explicitly disabled, custom proxied paths are not persisted in a usable way, and the new tests are not portable to Windows path semantics.

Findings

P1bd init --proxied-server always exits before doing any proxied setup

Where: cmd/bd/init.go:111

The new mode is exposed via --proxied-server, BEADS_DOLT_PROXIED_SERVER=1, no-CGO guidance, metadata support, and store creation code, but the init command immediately calls FatalError("--proxied-server is not yet implemented") whenever the flag or env var is set. That makes the primary user-facing entry point impossible to use and leaves the rest of the proxied-server implementation effectively unreachable for normal workspaces.

Impact: users following the new no-CGO error message or the new init flag cannot create a proxied-server workspace at all.

Suggested fix: remove the unconditional fatal guard once the path is ready, or keep the whole feature hidden from CLI/help/no-CGO messaging until it can initialize successfully. Add an integration or CLI test that runs bd init --proxied-server --non-interactive and proves metadata plus first open succeeds.

P1Custom proxied config/log/root flags are lost or resolved to the wrong place after restart

Where: cmd/bd/init.go:996-1005, internal/configfile/configfile.go:123-130, internal/configfile/configfile.go:419-442

init stores the raw flag values into metadata. On save, absolute values for dolt_proxied_server_config, dolt_proxied_server_log, and dolt_proxied_server_root_path are stripped. Relative values survive, but later resolution joins them under beadsDir, not the directory where the user supplied the flag. For example, bd init --proxied-server-config configs/server.yaml validates configs/server.yaml in the current working directory, but the next process looks for .beads/configs/server.yaml.

Impact: a workspace initialized with custom proxied paths silently falls back to defaults for absolute paths, or fails to reopen for relative paths that were valid during init.

Suggested fix: canonicalize custom path flags at init time into a stable representation before saving. Either copy generated/user config under .beads and persist a relative path there, or persist absolute paths only in a local, non-shared config layer. The resolver and save policy need to agree on the contract.

P2New proxied-server tests fail on Windows because Unix absolute paths are used as test fixtures

Where: cmd/bd/proxied_server_test.go:113, cmd/bd/proxied_server_test.go:609, internal/configfile/configfile_test.go:523, internal/configfile/configfile_test.go:574, internal/configfile/configfile_test.go:608, internal/configfile/configfile_test.go:660, internal/configfile/configfile_test.go:694, internal/configfile/configfile_test.go:746

The tests use paths like /etc/dolt/server.yaml and /var/log/beads/server.log as if they are absolute on every platform. On Windows, filepath.IsAbs("/etc/dolt/server.yaml") is false, so the production code treats those fixtures as relative and joins them under the temp .beads directory.

Impact: the new tests fail locally on Windows even under CGO_ENABLED=0. GitHub's Windows smoke check may not exercise these exact packages, but the repository should keep normal targeted package tests portable.

Suggested fix: generate platform-native absolute paths with filepath.Join(t.TempDir(), ...) or gate POSIX-only path expectations behind runtime checks.
