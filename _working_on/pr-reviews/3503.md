# PR #3503 Review: fix(history): respect --json flag when issue has no history

**Author**: seanmartinsmith
**Files**: `cmd/bd/history.go` (+7/-5), `cmd/bd/history_embedded_test.go` (+21/0)
**Verdict**: approve

---

## Summary

The PR fixes a genuine bug: `bd history <id> --json` emits prose ("No history found for issue X") instead of valid JSON when the issue has no history, breaking any pipeline that pipes output to `jq` or expects a consistent JSON contract. The fix is a minimal, correct reordering of the empty-history guard and the JSON output branch. The new test validates the fix using strict `json.Unmarshal` rather than the lenient helper that would silently mask the original bug.

---

## Strengths

- **Minimal, surgical change**: ~5 lines reordered, no new logic, no risk of regression to non-empty paths.
- **Consistent with codebase contract**: `bd list --json` and `bd show --json` already return valid JSON for empty results; this aligns history to the same pattern.
- **Test design is better than the helper**: The new subtest avoids the `bdHistoryJSON` helper's `strings.Index(s, "[")` lenient scan which returns `nil` for prose (masking the bug). Using strict `json.Unmarshal` on full stdout is the right call and the PR author correctly explains why.
- **Lint and all 11 subtests pass** per the PR description.

---

## Concerns

- **`--limit` applied before the empty-history check**: After the reorder, the `--limit` slice happens first (lines 41-43 in the post-patch file), then `outputJSON`, then the empty-check. For an empty slice this is harmless, but the ordering is slightly odd semantically — if `history` is empty the limit guard is a no-op so there is no correctness issue, just minor readability noise.
- **`outputJSON` wraps arrays through `wrapWithSchemaVersion`**: In legacy mode (default), arrays pass through unchanged (confirmed in `output.go`), so `[]` is emitted as `[]` not `{"schema_version":1,"data":[]}`. With `BD_JSON_ENVELOPE=1` the envelope wraps it. The new test does not exercise the envelope mode. This is a pre-existing gap, not introduced here, but worth noting.
- **Test uses `CombinedOutput`**: stdout and stderr are merged. If the binary ever emits a warning to stderr (e.g., the deprecation notice from `emitEnvelopeDeprecation`) this test would not fail, but a consumer parsing only stdout would also be fine. Low risk.
- **No test for `--limit --json` on empty history**: `bd history --json --limit 5 hi-nonexistent999` is untested. Trivially correct from code inspection but would add confidence.

---

## Suggestions

- `cmd/bd/history.go:38-43` (post-patch): Consider moving the `--limit` guard to after the empty-check for clarity, since applying a limit to an empty slice is meaningless. Not a correctness issue — just makes the intent more readable.
- `cmd/bd/history_embedded_test.go:189` (new subtest): Consider adding a second assertion that verifies `--json --limit 2 hi-nonexistent999` also returns `[]`, to cover the limit+json+empty combination.
