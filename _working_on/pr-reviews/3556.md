## Summary

A "trailing exec block" is the pattern used by tools like `pre-commit init-templatedir`: the hook file ends with an `if/elif/else/fi` ladder where every non-error branch replaces the current process via `exec <cmd>`. Because `exec` replaces the running shell, any shell code appended below it is permanently unreachable. Before this PR, `injectHookSection` unconditionally appended the `BEADS INTEGRATION` block at the bottom of preserved hooks, so on repos using pre-commit's templatedir mechanism `bd export` silently never fired on commits. The fix adds `findExecBlockInjectionPoint`, which walks backward from the end of the file to detect this pattern and returns the byte offset of the enclosing `if` (or bare `exec` line), so Case 3 of `injectHookSectionWithDepth` can splice the bd section in above the unreachable zone rather than below it.

- **Minimal footprint**: only Case 3 ("no markers") is touched; Cases 1 and 2 are untouched, and the fallback to append-at-end is preserved for every hook layout that doesn't end in `exec`.

- **Excellent test coverage**: table-driven unit tests for the helper plus four integration-style tests for `injectHookSection`, including the verbatim pre-commit template, a bare `exec`, a mid-file exec (must not trigger the new path), and a regression guard. `TestIsAllowedAfterExec_TableDriven` and `TestIsExecLine_TableDriven` cover all branches of the two small helpers.

The two PRs are independent (different code paths) and can be reviewed and merged separately without ordering constraints — neither introduces or depends on the other's changes.

## Concerns

### 1. Heredocs with `exec` inside — false positive (primary concern)

`findExecBlockInjectionPoint` splits on `\n` and applies `isExecLine` to the trimmed content of each line. A heredoc whose body happens to contain a line starting with `exec` will fool the scanner:

```sh
cat <<'EOF'
exec something
EOF
do-real-work
```

Here `exec something` is just text, not a shell statement. `isAllowedAfterExec` will return `false` for `do-real-work` (correct — falls back to append), but if `exec something` happened to be the *last* non-filler line before EOF the helper would return a non-negative offset and rewrite a hook it shouldn't touch. The fix does not need to be perfect — these are exotic hooks — but the PR description does not acknowledge the limitation, which matters for code reviewers and future maintainers. A doc comment noting the heredoc caveat would suffice; a full heredoc-aware parser is out of scope.

### 2. `strings.Split` drops the trailing newline — off-by-one risk

`strings.Split(content, "\n")` on a file ending in `\n` produces a spurious empty string as the last element. The backward scan in step 1 of `findExecBlockInjectionPoint` will hit this empty element first (classified as filler) and then continue, which is harmless in the happy path but means the last `lines[i]` examined in the byte-offset loop (step 3) may be that empty sentinel. For files that end without a trailing newline the behaviour differs: the last real line is what the scan sees first. This is unlikely to cause a real bug given how git hooks are written, but it is subtle enough to deserve a comment, and a `strings.TrimRight(content, "\n")` before splitting (or using `strings.Split(strings.TrimRight(...), "\n")`) would make the intent explicit.

### 3. Commented-out `exec` lines

`isExecLine` operates on `strings.TrimSpace(line)`. A line like `    # exec /legacy/path` is handled correctly (comment prefix is filtered by `isAllowedAfterExec`). However, a line like `exec /legacy/path  # currently disabled` would be treated as a live `exec` statement. This is obscure but worth a one-line note in the doc comment.

### 4. Multiple disjoint exec blocks

The PR description correctly states the scanner finds "the first effective exec from the bottom." If a file has two separate `if/exec` blocks neither of which is at the very tail (i.e. real code follows both), the backward scan hits the non-filler real code and returns `-1`, falling back to append. This is the correct and safe behaviour, but it is not covered by a test. The existing `TestInjectHookSection_ExecInMiddleStillAppends` covers an exec that is not at the tail; a complementary fixture with two disjoint exec blocks and real code after both would close this gap.

### 5. Idempotency — already handled, worth verifying explicitly

Once the bd section is injected above the `if` block, the file has markers. Re-running `bd hooks install` will hit Case 1 (marker replacement) rather than Case 3, so `findExecBlockInjectionPoint` is never called again. Idempotency is therefore inherited from the existing marker logic, not from this PR's code. This is fine, but an idempotency fixture (run `injectHookSection` twice on the pre-commit template) would make the guarantee explicit.

## Suggestions

- **`cmd/bd/hooks.go`, `findExecBlockInjectionPoint` doc comment** (around line 163 in the PR): add a sentence noting the heredoc caveat and the commented-out-exec edge case.
- **`cmd/bd/hooks.go`, step 1 of `findExecBlockInjectionPoint`** (around line 176): replace `strings.Split(content, "\n")` with `strings.Split(strings.TrimRight(content, "\n"), "\n")` to eliminate the spurious empty trailing element and make the off-by-one reasoning obvious.
- **`cmd/bd/hooks_exec_composition_test.go`**: add one table-driven case to `TestFindExecBlockInjectionPoint_TableDriven` for "two exec blocks with real code after both" to confirm the `-1` fallback. Add one idempotency test calling `injectHookSection` twice on the pre-commit template input.
