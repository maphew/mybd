## Summary

PR #3317 is directionally valuable: Beads should have a first-class Codex distribution path, and the reusable agent workflow belongs in a skill that can be packaged by a Codex plugin. The current PR is too broad to review or land safely because it combines at least six product decisions:

- Packaging the existing Beads skill as a shared Claude/Codex plugin tree.
- Adding Codex plugin metadata and a repo-local Codex marketplace entry.
- Teaching `bd setup codex` to install/check/remove a project-local skill.
- Adding optional Codex hooks, a hidden `bd run-hook --codex`, and compaction recovery state.
- Changing `bd init` defaults so skills/hooks are installed by default, including under `--skip-agents`.
- Moving or rewriting Claude command/resource docs from MCP-oriented guidance to CLI-first generic agent guidance.

The clean path is to keep the plugin/skill packaging work, split setup support from hooks, and defer `bd init` default behavior until the install surface has shipped and been tested independently.

---

## Keep

**1. Shared `plugins/beads` package as the distribution envelope.**  
Moving plugin assets under `plugins/beads` is the right shape if Beads wants one package that can expose Claude and Codex metadata while sharing the same skill content.

**2. Minimal Codex plugin manifest.**  
`plugins/beads/.codex-plugin/plugin.json` is the smallest useful first slice. It should declare the Beads plugin, point at `./skills/`, and avoid requiring hooks or setup-time side effects.

**3. Skill link/path cleanup.**  
Replacing `${CLAUDE_SKILL_DIR}` links in `SKILL.md` with relative paths is appropriate for a cross-agent skill. The skill should still state that `bd prime` is authoritative when generated skill guidance and runtime CLI guidance disagree.

**4. Version/release script updates for the new manifest path.**  
Updating release tooling to track both `plugins/beads/.claude-plugin/plugin.json` and `plugins/beads/.codex-plugin/plugin.json` belongs with the packaging slice.

**5. Test hardening that is unrelated but good.**  
The doctor Git config isolation and Dolt cloud-auth env cleanup look useful, but they should be extracted to their own PR because they are not part of Codex plugin packaging.

## Split

**Slice 1: Minimal plugin/skill packaging.**

Goal: make Beads installable as a Codex plugin that provides the Beads skill and no runtime mutation beyond what plugin installation normally does.

Scope:

- Add `plugins/beads/.codex-plugin/plugin.json`.
- Move or copy the existing Beads skill into `plugins/beads/skills/beads`.
- Preserve Claude plugin metadata under `plugins/beads/.claude-plugin/plugin.json` if the shared tree is desired.
- Update marketplace/release/version scripts for the new paths.
- Keep `AGENTS.md` project-local policy separate from the plugin.
- Keep `bd prime` as runtime source of truth.

Explicitly out of scope:

- No `.codex/hooks.json`.
- No `.codex/config.toml`.
- No `bd run-hook --codex`.
- No `bd init` default behavior change.
- No broad command/resource rewrites beyond path neutrality required for the moved skill.

**Slice 2: `bd setup codex` installs/checks/removes the skill/plugin cleanly.**

Goal: provide an explicit CLI setup path after the plugin package exists.

Scope:

- `bd setup codex` installs the Beads skill/plugin files into the project-local location chosen by the Codex plugin model.
- `bd setup codex --check` verifies expected files and versions.
- `bd setup codex --remove` removes only Beads-owned files and preserves user-owned `.agents` or `.codex` content.
- If AGENTS.md fallback support is kept, gate it behind an explicit flag such as `--agents`; do not make it implicit.

Review focus:

- File ownership boundaries.
- Idempotency.
- Exact remove behavior.
- Whether `.agents/skills/beads` is truly the right project-local install target for Codex, versus plugin-managed installation.

**Slice 3: Optional Codex hooks and compaction reinjection.**

Goal: add hooks only after the base plugin/skill is stable.

Scope:

- `bd setup codex --hooks` may write `.codex/hooks.json` and enable `features.codex_hooks`.
- `bd run-hook --codex` can handle `UserPromptSubmit` compaction recovery.
- Hook state such as `bd-run-hook-state.json` must be clearly ignored and should not require mutating the root `.gitignore`.
- Tests need to cover malformed hook JSON, existing user hooks, duplicate managed hooks, removal, and existing `.codex/config.toml` preservation.

Review focus:

- Codex hook schema stability.
- Avoiding duplicate reinjection after compaction.
- Avoiding silent modification/removal of user hook commands.
- Whether `SessionStart -> bd prime --codex` is still needed if Codex plugin/skill loading is sufficient.

**Slice 4: `bd init` defaults.**

Goal: consider default install behavior only after setup and hooks have landed separately.

Scope:

- Decide whether new repos should get a skill/plugin by default.
- Decide whether hooks should ever be default, or always explicit.
- Revisit `--skip-agents`, `--stealth`, `--skip-hooks`, and auto-staging semantics with the new surface area.

Recommended default: do not install Codex hooks from `bd init` in the first Codex release. `bd init` should remain conservative; hook installation should require `bd setup codex --hooks` until there is real usage feedback.

## Revert Or Defer

**Revert broad command/resource rewrites.**  
The author already noted that some moved command files were rewritten from "use MCP tool" to "use `bd ...`". Those rewrites should not ride along with packaging. For slice 1, preserve existing Claude command behavior unless a line must change because the file moved or because a path must be agent-neutral.

**Defer `bd init` installing `.agents` and `.codex` files by default.**  
This materially changes what `bd init` writes into new repositories and what gets auto-staged. That deserves a separate PR and explicit maintainer decision.

**Defer hidden hook runtime.**  
`cmd/bd/codex_hook.go` is substantial new runtime behavior. It is not needed to prove plugin packaging and should be reviewed independently after the base skill/plugin shape is accepted.

**Defer generic "Claude -> agent" doc renames unless they are required for the shared skill.**  
Renaming conceptual docs is lower-risk but creates noisy review surface and can accidentally weaken Claude-specific guidance.

**Extract unrelated test hardening.**  
Doctor Git config isolation and Dolt auth env cleanup should be their own small PRs so they can land without waiting on the Codex design.

## Proposed Upstream Comment

This is directionally the right area, but I would not try to land this PR as one unit. It combines plugin packaging, setup/install behavior, Codex hooks, compaction recovery, `bd init` defaults, docs moves, command rewrites, and unrelated test hardening. That makes the design hard to review and the regression surface too large.

Recommended split:

1. Land a minimal plugin/skill packaging PR first: `plugins/beads/.codex-plugin/plugin.json`, shared `plugins/beads/skills/beads`, relative skill links, and version/release script path updates. No hooks, no `bd init` changes, no hidden hook runtime.
2. Follow with `bd setup codex` support to install/check/remove the Beads skill/plugin explicitly. Keep AGENTS.md fallback behind an explicit flag if it remains.
3. Add optional hooks separately behind `bd setup codex --hooks`, including `bd run-hook --codex` and compaction recovery tests.
4. Revisit `bd init` defaults only after the base package and explicit setup path are stable.

Concrete keep/split guidance:

- Keep the Codex plugin manifest and shared skill packaging.
- Keep relative path cleanup in the skill and keep `bd prime` as the runtime source of truth.
- Split unrelated doctor/Dolt test hardening into its own PR.
- Revert broad command/resource rewrites that only change MCP-oriented wording to CLI-first wording unless the rewrite is required by the file move.
- Defer `.codex/hooks.json`, `.codex/config.toml`, auto-staging `.codex`/`.agents`, and `--skip-agents` semantic changes out of the first slice.

That first slice should be small enough to review on its own and would give Codex users immediate value without committing Beads to hook behavior or new `bd init` defaults prematurely.
