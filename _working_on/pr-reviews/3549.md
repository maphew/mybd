## Summary

When a user's working directory is a symlink (e.g. `~/src` → `/data/disk-b/src/`), `DirToFileURL` was emitting the symlink-form path verbatim in the `file://` URL. A Dolt SQL server running inside a container with a bind-mount of only the real path would then fail to locate the directory, returning a permission-denied error. The fix inserts `filepath.EvalSymlinks(abs)` between the existing `filepath.Abs` and URL assembly; if resolution fails (dangling symlink, race-deleted target) the function falls through to the un-resolved absolute path, so callers never see a new error class.

---

## Strengths

- **Minimal surface area**: single-file functional change (3 lines), no new imports, no new dependencies.
- **Correct error-tolerance design**: silently falling back rather than propagating `EvalSymlinks` errors avoids introducing new failure modes for callers that currently pass dangling or non-existent paths.
- **Well-structured tests**: three regression tests with clear names cover the happy path (regression), the broken-symlink fallback, and the no-op round-trip. The macOS `/tmp → /private/tmp` trap is explicitly handled in the test by evaluating the expected real path via `EvalSymlinks` rather than hard-coding it.
- **Good PR description**: commit strategy, manual smoke-test, and pre-existing unrelated failures are all documented. Reviewer can isolate the test commit and observe a red-to-green transition.

## Concerns

### 1. Symlink semantics change for users who rely on symlink paths (main concern)

`DirToFileURL` is called from four call sites (two in `dolt/store.go`, two in `embeddeddolt/version_control.go`) for both `BackupDatabase` and `RestoreDatabase`. After this patch, the stored/used URL is always the realpath. Any user who deliberately keeps the backup directory behind a stable symlink and relies on the symlink path being recorded in Dolt's remote config (so the remote is portable via that alias) will silently get the resolved path instead. This is probably fine in practice, but the doc comment does not mention this trade-off — it only describes the container/bind-mount motivation.

### 2. Windows behavior

`filepath.EvalSymlinks` on Windows resolves Windows junction points and reparse points. The behavior is documented as best-effort. If a Windows user has intentional junction-based layouts, the symlink resolution will silently rewrite their paths. Low probability, but worth a comment in the doc string or a TODO.

### 3. TOCTOU / permission races

`EvalSymlinks` opens the filesystem at call time. If the symlink is valid at the time of the call but the target is removed before the Dolt server uses the URL, the URL will point to a now-missing real path rather than the (still-existing) symlink. This is a narrow race and was possible pre-patch too (the directory could disappear), but the fallback contract makes this window slightly wider. Not a blocker.

### 4. Silent fallback could mask misconfiguration

If `EvalSymlinks` returns an error for a reason *other* than a dangling symlink (e.g. permissions), the function silently falls back to the symlink path and the downstream Dolt error surfaces instead. The caller gets a confusing "permission denied from Dolt server" error rather than "could not resolve path". Consider logging or wrapping the `EvalSymlinks` error at `warn` level before falling back.

### 5. Test: BrokenSymlinkFallsBack is slightly over-specified

`backup_test.go:88` — the test asserts `got == "file://" + filepath.Abs(link)`. On macOS where `t.TempDir()` itself is behind `/tmp → /private/tmp`, `filepath.Abs` does *not* resolve through the symlink, so the assertion correctly compares symlink-path to symlink-path. However, if the OS resolves `filepath.Abs` differently in future, the test could become brittle. The comment in the test acknowledges this, so it's a nit.
