# PR Review: #3482 — fix(export): sort memory keys for deterministic JSONL output (GH#3474)

**Author:** kevglynn
**Verdict:** approve-with-nits

---

## Summary

This PR fixes non-deterministic ordering of `_type:"memory"` lines in exported JSONL output by collecting matching keys from a `map[string]string` into a slice, sorting alphabetically, then iterating in order. The root cause is standard Go map iteration randomization. The fix is correct and matches the approach suggested in the linked issue (#3474). However, the fix is **incomplete**: `export_auto.go` has an identical code path at line 210 that is not touched, meaning the pre-commit hook path (which the issue explicitly calls out) still produces non-deterministic output.

---

## Strengths

- Minimal, targeted change — only the affected loop is touched; no unrelated reformatting.
- Standard `sort.Strings` is the idiomatic Go solution and consistent with the rest of the codebase (no `maps.Keys` or `slices.Sorted` patterns appear in cmd/bd).
- The condition inversion (from `!HasPrefix → continue` to `HasPrefix → append`) makes the intent clearer.
- Comment cites the issue number for traceability.
- The fix is logically correct for `export.go`; sorting by storage key is equivalent to sorting by user-visible key since `fullPrefix` is a constant offset shared by all entries.

---

## Concerns

1. **Incomplete fix (correctness — high):** `export_auto.go:210` contains an exact duplicate of the unfixed loop. The linked issue (#3474) explicitly names both files and lines as affected. The pre-commit hook calls the auto-export path, so this is the highest-frequency trigger of the bug. The PR test plan even mentions "bd export twice" but not the auto-export path.

2. **No regression test added:** The PR description acknowledges that existing determinism tests miss this case, and suggests the exact repro (seed N≥3 memories, export twice, assert byte-identical output), but no test was added. Without it, this can silently regress.

3. **Minor — error handling in `export_auto.go`:** The auto-export memory loop already silently swallows `json.Marshal` errors (`continue` on error). This is pre-existing behaviour, but the incomplete fix leaves that path unfixed while the corrected `export.go` version has proper error propagation.

---

## Suggestions

- **`cmd/bd/export_auto.go:210`** — Apply the same sort fix here. The block at lines 210–231 is structurally identical to the `export.go` loop before this PR. A refactor into a shared helper (e.g., `writeMemoryRecords(w io.Writer, allConfig map[string]string, fullPrefix string)`) would eliminate the duplication entirely and prevent future divergence.
- **`cmd/bd/export_test.go` or `cmd/bd/export_embedded_test.go`** — Add a determinism test: seed 3+ memories, call `runExport` twice to separate temp files, assert byte-for-byte identical output.
- Consider Go 1.26's `slices.Sorted(maps.Keys(allConfig))` as a one-liner if the project ever adopts the `maps`/`slices` stdlib packages — currently neither is used in cmd/bd, so `sort.Strings` is the right call for now.

---

## Test Coverage

- Existing tests (`TestExportToFile`, `TestExportToStdout`, `TestExportScrub`) cover basic export correctness but create no memories and do not assert output stability across multiple runs.
- The auto-export path has one test (`export_auto_test.go`) focused on a git worktree staging bug (GH#3311) — no memory ordering coverage.
- **Missing:** any test asserting byte-identical output on repeated exports when memories are present, in either code path.
