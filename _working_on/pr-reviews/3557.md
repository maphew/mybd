# PR #3557 Review: fix(hooks): preserve user content when migrating v0.62.x → v1.0.x (GH#3536)

**Author**: pmgledhill102 (Paul Gledhill)
**Verdict**: approve-with-nits
**Stats**: +190/-3, 2 files changed (`cmd/bd/hooks.go`, `cmd/bd/hooks_marker_skip_test.go`)

---

## Summary

In v0.62.x, `bd` injected its BEADS INTEGRATION block *into* user-owned hook files rather than replacing them. When users later upgraded to v1.0.x (which relocates hooks to `.beads/hooks/`), `preservePreexistingHooks` would see the begin-marker and treat the entire file as bd-managed, silently discarding the user's dispatcher, framework integration, or custom logic above the block. This PR fixes that by stripping the bd block via the existing `removeHookSection` helper before preserving the file, skipping only when stripping leaves nothing meaningful (checked by the new `isOnlyShebangOrEmpty` helper). The data-loss bug was reproducible on 8 of 10 personally-verified real repos.

---

## Strengths

- The fix is minimal and surgically correct. The `inlineHookMarker` skip path (wholly bd-owned one-liners) is deliberately preserved; only the `hookSectionBeginPrefix` path changes.
- Reuses `removeHookSection`, which already handles all four marker states (valid pair, orphaned BEGIN, orphaned END, reversed markers). No new parsing logic is needed or introduced.
- `isOnlyShebangOrEmpty` is a pure, side-effect-free helper with a clear contract, making it easy to reason about and test.
- Test coverage is thorough for a targeted fix: 8 table-driven cases for the helper, the exact GH#3536 repro fixture, both wholly-managed inverse cases, and a no-markers regression guard. The PR description shows tests run and lint passes.
- The sibling PR (#3556) addresses a distinct code path (`injectHookSection` placing the block below an `exec` line) with no shared logic, so there is no ordering hazard between the two.
- Comments in the changed lines explain the historical *why* (GH#1380, GH#1120 lineage), which will prevent the pattern from being collapsed back by a future cleanup.

## Concerns

**1. CRLF-in-v0.62.x files (minor, data-quality).**
The new code path calls `removeHookSection(contentStr)` and then writes `contentStr` directly (line ~744 in the patched file). On Windows or in repos with `autocrlf=true`, a v0.62.x hook file can have `\r\n` line endings. `removeHookSection` splits on `"\n"` and uses `strings.LastIndex(..., "\n")` for boundary math; a trailing `\r` left on each preserved line would end up written into `.beads/hooks/<name>`. The non-husky write path lacks the `strings.ReplaceAll(newContent, "\r\n", "\n")` normalization that `injectBeadsSection` applies (hooks.go:622). This is a pre-existing gap that the PR's new strip-then-preserve path inherits rather than introduces, but since the PR is touching exactly this code it is a good moment to normalize. Adding `contentStr = strings.ReplaceAll(contentStr, "\r\n", "\n")` just before the `isOnlyShebangOrEmpty` check (after the strip) would close it with one line.

**2. `isOnlyShebangOrEmpty` classifies bare `#!` comments-only files as empty (intended but worth noting).**
The helper treats any `#`-prefixed line as a comment, which means a file like `#!/bin/sh\n# handwritten preamble comment\n` is classified as "nothing to preserve" and skipped. That seems correct by the PR's intent (comments alone aren't user logic), but a human reviewer might expect a file with a non-shebang comment like `# DO NOT EDIT — regenerated by CI` to be preserved as evidence. The PR description does not call this out. This is a design choice, not a bug, but worth a comment in the helper's godoc noting that non-shebang comments are intentionally treated as non-content.

**3. `removeHookSection` drops trailing blank lines with a three-newline check; no test covers trailing-blank preservation edge.**
After stripping, `removeHookSection` collapses runs of `\n\n\n` → `\n\n`. If user content ends in two blank lines followed by the bd block, the strip may over-trim the user content. This is subtle and pre-existing, but the new test fixture in `TestPreservePreexistingHooks_StripsLegacyMarkerBlock` uses exactly one blank line separator — adding a variant with two blank lines before the marker would give confidence this boundary works.

## Suggestions

- `cmd/bd/hooks.go`, in the new `hookSectionBeginPrefix` branch (around line 750 in the patched file): add `contentStr = strings.ReplaceAll(contentStr, "\r\n", "\n")` after `contentStr = stripped` to normalize CRLF before the write. One line, consistent with the existing normalization at line 622.
- `cmd/bd/hooks.go`, `isOnlyShebangOrEmpty` godoc: add a one-sentence note: "Non-shebang comment lines (e.g. # preamble) are treated as non-content and do not prevent the 'empty' classification." This makes the intent explicit to future maintainers.
- `cmd/bd/hooks_marker_skip_test.go`, `TestPreservePreexistingHooks_StripsLegacyMarkerBlock`: add a second fixture variant with two blank lines before the `# --- BEGIN` marker to exercise `removeHookSection`'s blank-line-trim boundary.
